# OpenSpec Roadmap для MassEditor

## Цель

Этот документ нужен как локальный рабочий ориентир для репозитория `MassEditor`.
Он описывает, как здесь лучше использовать OpenSpec от первичного анализа до завершения change, чтобы не смешивать спецификацию, задачи и код без явного шага.

Репозиторий является backend-only плагином Shop-Script `masseditor`, поэтому OpenSpec здесь должен поддерживать именно backend-сценарии, безопасность, TDD и проверку совместимости с Webasyst/Shop-Script.

## Текущий статус

По текущему checkout OpenSpec-root здоров:

- `openspec doctor` сообщает, что root `healthy: true`.
- `openspec list --json` возвращает `changes: []`, то есть активных changes сейчас нет.
- `openspec list --specs --json` показывает 5 capability-specs:
  - `backend-ui-i18n`
  - `mass-operations`
  - `plugin-architecture`
  - `product-selection`
  - `quality-workflow`

Фактическая структура на диске:

- `openspec/config.yaml`
- `openspec/specs/backend-ui-i18n/spec.md`
- `openspec/specs/mass-operations/spec.md`
- `openspec/specs/plugin-architecture/spec.md`
- `openspec/specs/product-selection/spec.md`
- `openspec/specs/quality-workflow/spec.md`
- `openspec/changes/archive/`

Рабочих `openspec/changes/*` сейчас нет.

## Когда какую команду использовать

| Команда | Когда использовать | Что она дает |
| --- | --- | --- |
| `openspec doctor` | Перед началом работы или если поведение root кажется странным | Быстрая диагностика корня, ссылок и здоровья OpenSpec |
| `openspec list` | Когда нужно понять, какие changes сейчас активны | Список changes по умолчанию |
| `openspec list --specs` | Когда нужно увидеть набор capability-specs | Инвентаризация доступных спецификаций |
| `openspec show <item>` | Когда нужно прочитать конкретный change или spec | Полный текст выбранного артефакта |
| `openspec context` | Когда нужно собрать рабочий контекст для выполнения | Подготовленный контекст для следующего шага |
| `openspec instructions` | Когда нужно понять, как OpenSpec ожидает вести работу | Подсказки по процессу и артефактам |
| `openspec templates` | Когда нужно понять структуру шаблонов change/spec/task | Пути и шаблоны для генерации артефактов |
| `openspec validate --strict --all` | Перед завершением любого change | Жесткая финальная проверка всех OpenSpec-артефактов |

## Рекомендуемый процесс работы

### 1. Сначала понять, есть ли уже change

Для любой новой задачи сначала проверьте:

```bash
openspec doctor
openspec list
openspec list --specs
```

Если активный change уже существует и задача к нему относится, продолжайте его, а не создавайте новый параллельно.

### 2. Если задача сформулирована не до конца, используйте `explore`

`openspec explore` уместен, когда:

- границы задачи еще плавают;
- непонятно, в какой capability-spec она попадает;
- нужно уточнить риски, non-goals и зависимости;
- задача может затронуть несколько областей и нужно сначала разложить ее по полкам.

Для MassEditor это особенно полезно, когда запрос касается:

- массовых операций и ограничений на изменения;
- выбора товаров и серверной фильтрации;
- backend UI и локализации;
- прав доступа, CSRF, SQL-безопасности или N+1.

### 3. Если задача уже ясна, используйте `propose`

`openspec propose` подходит, когда:

- есть понятный пользовательский эффект;
- известны границы изменения;
- можно сразу зафиксировать требования, tasks и design;
- нужен формальный change до изменения кода.

Для этого репозитория proposal лучше вести на русском и явно связывать его с существующими specs:

- `plugin-architecture`
- `product-selection`
- `mass-operations`
- `backend-ui-i18n`
- `quality-workflow`

### 4. Когда уместен `apply`

`openspec apply` нужен, когда change уже одобрен и спецификация готова к реализации.

В MassEditor это означает:

- сначала есть change с понятной целью;
- задачи разбиты на небольшие проверяемые шаги;
- есть понимание, какие тесты писать первыми;
- ясно, какие файлы кода можно трогать.

Важно: `apply` не должен превращаться в прямую правку кода без актуальной спецификации.

### 5. Когда делать `sync`, а когда `archive`

`sync` используйте, когда:

- delta-specs уже обновлены в change;
- нужно слить их в main specs;
- сам change еще не завершен полностью или его артефакты нужно сохранить для дальнейшей работы.

`archive` используйте, когда:

- change завершен;
- код реализован;
- проверки пройдены;
- main specs нужно привести в соответствие с итоговым состоянием;
- change больше не нужен как рабочая единица.

Практически для MassEditor это означает:

- `sync` для промежуточного обновления спецификаций;
- `archive` для закрытия законченного изменения.

### 6. Когда использовать `status`

`openspec status` полезен вместо ручной проверки, если нужно быстро понять:

- на какой стадии change;
- что уже сделано;
- что еще осталось;
- есть ли блокирующие проблемы.

Это лучше, чем вручную сравнивать файлы, когда change уже ведется и нужно быстро сориентироваться.

## Что лучше подходит для чего

- `doctor` — диагностика root и зависимостей.
- `list` и `show` — инвентаризация и чтение артефактов.
- `context` — сбор рабочей области для выполнения.
- `explore` — когда задача еще не до конца сформулирована.
- `propose` — когда нужен формальный change до кода.
- `apply` — когда спецификация уже одобрена и можно реализовывать.
- `sync` — когда нужно слить delta-specs в main specs без финальной архивации.
- `archive` — когда change завершен и спецификации нужно обновить.
- `validate` — финальный gate перед завершением.
- `instructions` и `templates` — когда нужно понять структуру артефактов и следующие шаги.

## Проверки и завершение

Перед закрытием любой OpenSpec-задачи в этом репозитории выполните:

1. `openspec validate --strict --all`
2. Проверки, соответствующие затронутому коду:
   - для PHP: `php -l` по измененным файлам и `bash tests/run-php-tests.sh`
   - для JS: `node --check wa-apps/shop/plugins/masseditor/js/masseditor.js` и `bash tests/run-js-tests.sh`
3. Если менялись только OpenSpec-артефакты или документация, достаточно OpenSpec-валидации, если runtime-код не трогался.

## Примечания по этому репозиторию

- Все OpenSpec proposal/spec/task/report артефакты для MassEditor должны быть на русском языке.
- Перед изменениями в Webasyst/Shop-Script нужно сверяться с официальной документацией или локально подтвержденным кодом.
- Перед проектированием и реализацией любых изменений всегда сначала проверяйте актуальные рекомендации Webasyst/Shop-Script и опирайтесь на них как на базовый источник решений.
- Для каждого нового или измененного пользовательского backend-текста, включая toast-уведомления, ошибки валидации и `aria-label`, обязательно указывайте источник i18n-ключа и проверяйте полный текст на `ru_RU` и `en_US` согласно `interface_language`. Английский fallback в JS допустим только как защитный путь, а не как штатный текст русского интерфейса.
- Для этого репозитория базовый workflow уже задан существующими capability-specs, поэтому не стоит создавать новый change без понимания, к какой из них относится задача.
- Не смешивайте OpenSpec-артефакты и кодовые изменения без явного шага `apply` или `archive`.
- Для backend-only плагина особенно важно сохранять:
  - тонкие actions;
  - логику в `lib/classes/`;
  - модели в `lib/models/`;
  - локализацию через i18n-слой;
  - серверную валидацию, CSRF и безопасный SQL.

## Краткий чеклист для MassEditor

- Сначала определить, есть ли уже подходящий change.
- Если задачи еще не хватает ясности, идти через `explore`.
- Если задача ясна, оформлять `propose`.
- Перед кодом держать TDD и проверять релевантные docs.
- Для видимого текста указать ключ i18n, переводы `ru_RU`/`en_US` и тесты обоих языков.
- После реализации обязательно запускать `validate`.
- Закрывать change только после синхронизации или архивации, если это действительно финал.
