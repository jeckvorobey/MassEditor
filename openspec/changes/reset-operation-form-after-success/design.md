## Context

`shopMasseditorPluginBackendAction::execute()` создаёт дефолтный массив `$operation_form`. Сейчас после успешного применения он снова объединяется с нормализованным request сервиса, поэтому шаблон получает применённые параметры. При ошибке `restorePostStateAfterError()` намеренно сохраняет POST-состояние.

## Goals / Non-Goals

**Goals:**

- После подтверждённого успешного применения отрисовывать дефолтное состояние формы.
- Сохранить текущую возможность исправить форму после ошибки.
- Закрепить оба пути PHP-регрессионным тестом.

**Non-Goals:**

- Не менять сервис массовых операций, выбор товаров, журнал, CSRF, права, валидацию, JavaScript и локализацию.
- Не очищать фильтры каталога и не изменять данные в БД помимо уже выполняемой операции.

## Decisions

- Action сохранит исходный массив `$operation_form` до обработки POST и восстановит его только после успеха. Это минимально сохраняет тонкий controller, единый дефолтный контракт для шаблона и не дублирует дефолты в JavaScript.
- Путь exception останется без изменений: `restorePostStateAfterError()` продолжит объединять POST с дефолтами. Альтернатива — сбрасывать форму всегда — отклонена, поскольку лишила бы пользователя введённых данных при ошибке.
- Регрессионная проверка будет запускать настоящий action с успешной валидной операцией и проверять назначенный шаблону массив. Это проверяет пользовательский результат вместо внутренней детали реализации.

## Risks / Trade-offs

- [Неполное тестовое окружение action] → использовать уже применяемые в PHPUnit фейки Webasyst и минимальные валидные данные операции.
- [Непреднамеренное изменение error-flow] → отдельная проверка существующего сохранения POST при ошибке и ограниченный diff action.
- [Нет влияния на запросы] → изменение выполняется после применения и не добавляет SQL, циклов или загрузок.
