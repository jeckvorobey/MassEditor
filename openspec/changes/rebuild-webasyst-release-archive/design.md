## Context

Плагин публикуется как `tar.gz`, а production-дерево уже отделено от тестов и документации в `wa-apps/shop/plugins/masseditor/`. Существующий `dist/masseditor-1.1.0.tar.gz` имеет правильную одно-корневую структуру, но собран раньше текущего состояния: прямое сравнение распакованного архива с production-деревом показывает различия в 14 файлах.

Webasyst Store требует архив с единственной корневой директорией, названной по ID продукта, корректные `vendor` и трехчастную версию в `plugin.php`, UTF-8 для текстовых файлов, `.htaccess` в `lib/`, `templates/`, `locale/`, открытую серверную часть и отсутствие уязвимостей. Для обновления с gettext-локализацией официальная документация также требует актуальных `.po` и `.mo`.

## Goals / Non-Goals

**Goals:**

- Собрать актуальный `dist/masseditor-1.1.0.tar.gz` только из production-дерева плагина.
- Не заменить предыдущий архив, пока кандидат не пройдет структурные, metadata-, localization- и content-проверки.
- Подтвердить, что распакованный кандидат эквивалентен исходному дереву и пригоден для передачи в Webasyst Store.
- Оставить проверяемый итоговый отчет с размером и SHA-256 архива.

**Non-Goals:**

- Не менять номер версии `1.1.0` и не создавать новый функциональный релиз.
- Не менять PHP/JS/CSS/Smarty-поведение, схему БД, install/uninstall или Store-описание.
- Не включать материалы разработки и публикации внутрь продукта.
- Не загружать архив в Webasyst Store и не выполнять релиз во внешней системе.
- Не добавлять новый постоянный packaging framework или внешнюю runtime-зависимость плагина.

## Decisions

### 1. Использовать production-дерево как единственный manifest

Кандидат собирается копированием всей директории `wa-apps/shop/plugins/masseditor/` во временную `masseditor/` с сохранением скрытых `.htaccess`. До копирования источник проверяется на пустые директории, временные файлы, отладочные артефакты и неожиданные ссылки. Такой подход исключает ручной список файлов, который легко забыть обновить при появлении нового production-controller или ресурса.

Альтернатива — упаковать весь репозиторий с большим списком исключений. Она отклонена, потому что повышает риск попадания `tests/`, `docs/`, `openspec/`, Docker и локальных настроек.

### 2. Получать имя архива из проверенной версии плагина

Версия читается PHP из `lib/config/plugin.php`; сборка продолжается только для трехчастной версии и ожидаемого значения `1.1.0`. Там же проверяются числовой `vendor`, ID директории `masseditor` и наличие `lib/config/shop_support.json` с декларацией PREMIUM. Итоговое имя формируется как `dist/masseditor-<version>.tar.gz`.

Альтернатива — захардкодить имя независимо от `plugin.php`. Она отклонена, потому что допускает расхождение метаданных и файла, загружаемого в Store.

### 3. Синхронизировать gettext до формирования staging-копии

Для каждой локали `ru_RU` и `en_US` актуальный `.po` компилируется gettext-совместимым `msgfmt` в соответствующий `.mo` с включенной проверкой формата. Если проверенный компилятор недоступен, сборка останавливается, а старый архив не заменяется. Сгенерированные `.mo` входят в production-дерево и затем проверяются вместе с остальными файлами.

Альтернатива — взять существующие `.mo` без проверки. Она отклонена: текущие `.po` изменялись, а Webasyst использует бинарные каталоги переводов в runtime.

### 4. Собирать и проверять кандидат до атомарной замены

Архив сначала создается под временным именем вне финального пути. Для кандидата выполняются `gzip -t`, чтение `tar` manifest, проверка единственного верхнего каталога `masseditor/`, запрет абсолютных путей и `..`, проверка обязательных файлов и отсутствие материалов репозитория. Затем кандидат распаковывается во вторую временную директорию и рекурсивно сравнивается с исходным production-деревом.

Только после успешных проверок кандидат атомарно перемещается в `dist/masseditor-1.1.0.tar.gz`. Это сохраняет предыдущий рабочий архив при любой ошибке до финального шага.

### 5. Отделить упаковочную проверку от runtime-изменений

Красная проверка фиксирует, что старый архив не совпадает с текущим деревом. После пересборки та же проверка должна стать зеленой. Дополнительно перед упаковкой выполняются существующие PHP/JS-тесты, syntax checks и compliance-поиски; новый PHP или JavaScript для упаковки не добавляется, поэтому отдельные unit-тесты runtime-кода не требуются.

## Risks / Trade-offs

- [В архив попадет непреднамеренный production-файл] → До сборки просмотреть полный manifest источника и проверять запрещенные имена/типы; единственным источником остается узкая директория плагина.
- [Старый архив будет потерян при неудачной проверке] → Создавать отдельный кандидат и заменять финальный файл только после всех проверок.
- [`.po` и `.mo` разойдутся] → Обязательная компиляция обеих локалей проверенным `msgfmt`; отсутствие инструмента является стоп-условием.
- [Архив формально корректен, но содержит сломанный runtime] → До упаковки запускать PHP/JS suites, syntax checks и проверку production-файлов на debug-код.
- [Права или скрытые `.htaccess` потеряются при staging] → Копировать дерево с сохранением файлов и явно проверять `.htaccess` как в источнике, так и в распакованном архиве.
- [Локальные незакоммиченные изменения попадут в релиз] → Это ожидаемо только для текущего завершенного `1.1.0`; перед сборкой зафиксировать `git status` и подтвердить, что source manifest ограничен `wa-apps/shop/plugins/masseditor/`.

## Migration Plan

1. Зафиксировать failing-сравнение старого архива с текущим production-деревом.
2. Выполнить preflight-проверки, тесты и компиляцию локалей.
3. Создать и полностью проверить временный кандидат.
4. Заменить `dist/masseditor-1.1.0.tar.gz` только после зеленых проверок.
5. Повторно распаковать финальный архив, проверить manifest/эквивалентность и вывести размер с SHA-256.

Откат: если финальная повторная проверка неожиданно завершится ошибкой, вернуть предыдущий архив из временной копии текущего запуска и не считать задачу завершенной. Runtime-код и данные пользователей миграции не требуют.
