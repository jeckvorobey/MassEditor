## Context

Плагин `masseditor` выполняет массовые операции через backend-сервис `shopMasseditorPluginMassOperationService`. Операция `stock` уже принимает `stock_id`, но текущая ветка применения складской операции раньше зависела от наличия складских строк у товара. Из-за этого выбранный склад не всегда приводил к записи в `shop_product_stocks`.

После исправления этой ветки массовая запись сформирована как `REPLACE INTO shop_product_stocks (sku_id, stock_id, count)`. На Docker-стенде с реальной схемой Shop-Script таблица содержит обязательное поле `product_id` без значения по умолчанию. Поэтому такая запись завершается ошибкой MariaDB и откатывает всю операцию.

Официальная документация Shop-Script по хранению товаров фиксирует два режима остатков:

- если у товара есть хотя бы одна строка в `shop_product_stocks`, товар использует детализацию по складам;
- в складском режиме `shop_product_skus.count` и `shop_product.count` являются агрегатами;
- после изменения складских строк агрегаты исправляются через `shopProductModel->correct(product_id)`.

## Goals / Non-Goals

**Goals:**

- Сделать `stock_id > 0` главным признаком складской операции.
- Записывать только выбранный склад, не материализуя остальные склады.
- Сохранить прежний отказ для складских товаров в режиме `stock_id = 0`.
- Покрыть поведение PHP-тестами до реализации.

**Non-Goals:**

- Не менять backend UI, JavaScript, локализацию, фильтр товаров и схему БД.
- Не добавлять журнал `shop_product_stocks_log` в рамках этого исправления.
- Не менять публичный контракт массовой операции кроме уточненной обработки `stock_id`.

## Decisions

1. `stock_id > 0` всегда ведет в складскую ветку.

   Альтернатива: продолжать проверять наличие существующих строк через `usesWarehouseStockAccounting()`. Это сохраняет баг для товаров, у которых складских строк еще нет, поэтому отклонено.

2. Отсутствующая строка выбранного склада считается старым значением `0`; для склада с учетом материализуются нулевые строки всех настроенных складов.

   Обоснование: `shopProductModel->correct()` загружает все настроенные склады и заполняет отсутствующие строки `NULL`. При наличии хотя бы одного `NULL` Shop-Script считает общий остаток бесконечным. Нулевые строки явно означают нулевой остаток и позволяют ядру рассчитать общий числовой итог.

   Альтернатива: создавать строку только выбранного склада. Отклонено: при нескольких складах отсутствующие строки интерпретируются ядром как бесконечный остаток, и таблица показывает `∞` вместо общего значения.

3. Пакетная запись передает `product_id` для каждой строки склада.

   Альтернатива: передавать только `sku_id`, `stock_id` и `count`. Она отклонена: в фактической схеме Shop-Script `product_id` является обязательным полем.

   Альтернатива: вызывать `shopProductStocksModel::set()` для каждого SKU. Она отклонена для массовой операции, потому что модель отдельно читает существующую строку каждого SKU, создавая N+1 запросов. Сервис уже получил `product_id` и текущие остатки пакетно, поэтому добавление `product_id` в существующую параметризованную пакетную запись сохраняет производительность и идемпотентность.

4. После складской записи вызывается `shopProductModel->correct(product_id)`.

   Альтернатива: вручную обновлять `shop_product_skus.count`. Это противоречит модели Shop-Script, где SKU count в складском режиме является агрегатом.

## Risks / Trade-offs

- [Risk] `shop_product_stocks` может иметь дополнительные поля в конкретной версии Shop-Script. → Mitigation: использовать только подтвержденные поля `sku_id`, `stock_id`, `product_id`, `count`.
- [Risk] SQL unit-тест с fake-моделью не отражает обязательность полей реальной БД. → Mitigation: проверить состав параметров unit-тестом и выполнить сценарий на Docker-стенде с фактической схемой Shop-Script.
- [Risk] `REPLACE` перезаписывает строку выбранного склада. → Mitigation: операция должна устанавливать итоговый остаток выбранного склада, другие склады не затрагиваются.
- [Risk] Тестовый double модели может не фиксировать вызов `correct()`. → Mitigation: покрыть прямые записи и сохранить syntax/full PHP test run.
- [Risk] Пользователь выбирает «Без склада» для товаров со складским учетом и узнает об ограничении только после POST. → Mitigation: до открытия окна подтверждения подсвечивать селект склада и показывать существующее локализованное уведомление.
