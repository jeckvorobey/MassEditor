## MODIFIED Requirements

### Requirement: Поддерживаемые операции
Сервис MUST поддерживать только whitelisted операции: `price`, `compare_price`, `visibility`, `availability`, `description`, `tags`, `url`, `stock`, `features`, `categories`, `video`.

#### Scenario: Неизвестная операция
- **WHEN** request содержит operation вне whitelist
- **THEN** сервис MUST вернуть локализованную ошибку неизвестной операции

#### Scenario: Цена и compare price
- **WHEN** выполняется операция цены или compare price
- **THEN** сервис MUST поддерживать режимы `set`, `add`, `subtract`, `increase_percent`, `decrease_percent`, округление 0-4 знака и запрет отрицательного результата

#### Scenario: Описание, теги и URL
- **WHEN** выполняются операции контента
- **THEN** сервис MUST валидировать непустые значения, поддерживать режимы описания `replace`/`prepend`/`append`, теги `add`/`remove`/`replace` и URL `regenerate`/`template`

### Requirement: Остатки, характеристики и категории
Операции остатков, характеристик и категорий MUST соблюдать ограничения безопасности релиза.

#### Scenario: Остатки без выбранного склада
- **WHEN** операция stock не указывает склад или передает `stock_id = 0`
- **THEN** сервис МОЖЕТ менять общий SKU count только для товаров без складского учета и ДОЛЖЕН отклонить товары, у которых есть любые строки в `shop_product_stocks`

#### Scenario: Остатки выбранного склада
- **WHEN** операция stock передает `stock_id > 0`
- **THEN** сервис MUST записывать итоговый остаток только в `shop_product_stocks` для выбранного склада вместе с корректным `product_id` и MUST NOT напрямую сохранять новое значение в `shop_product_skus.count`

#### Scenario: Новый складской остаток товара
- **WHEN** операция stock передает `stock_id > 0`, а у SKU товара нет строки выбранного склада
- **THEN** сервис MUST считать старое значение выбранного склада равным `0`, MUST создать строку выбранного склада и MUST создать нулевые строки остальных настроенных складов для корректного пересчета общего остатка

#### Scenario: Режимы складского остатка
- **WHEN** операция stock передает `stock_id > 0`
- **THEN** сервис MUST поддерживать режимы `set`, `increase`, `decrease` и `infinite`, где `set` задает значение, `increase` прибавляет к старому значению выбранного склада, `decrease` запрещает отрицательный итог, а `infinite` записывает `NULL`

#### Scenario: Пересчет агрегатов после складского остатка
- **WHEN** операция stock успешно записывает остаток выбранного склада
- **THEN** сервис MUST обновить `shop_product.edit_datetime`, MUST пересчитать агрегаты товара через механизм Shop-Script и MUST отобразить общий числовой остаток вместо `∞`, если все складские остатки числовые

#### Scenario: Одиночные характеристики всех документированных типов
- **WHEN** операция features устанавливает значение одиночной корневой характеристики
- **THEN** сервис ДОЛЖЕН принимать формат значения, документированный Shop-Script для её типа, и ДОЛЖЕН сохранять только выбранную характеристику через записываемое свойство `features` объекта `shopProduct`

#### Scenario: Множественная характеристика
- **WHEN** выбрана корневая характеристика с `multiple=1`
- **THEN** сервис ДОЛЖЕН принимать дедуплицированный массив существующих `feature_value_ids[]` и передавать итоговые значения в Shop-Script как массив

#### Scenario: Режим replace
- **WHEN** операция множественной характеристики выполняется в режиме `replace`
- **THEN** сервис ДОЛЖЕН заменить только значения выбранной характеристики на переданный набор

#### Scenario: Режим add
- **WHEN** операция множественной характеристики выполняется в режиме `add`
- **THEN** сервис ДОЛЖЕН сохранить существующие значения выбранной характеристики и добавить только отсутствующие выбранные значения

#### Scenario: Режим remove
- **WHEN** операция множественной характеристики выполняется в режиме `remove`
- **THEN** сервис ДОЛЖЕН удалить только выбранные значения выбранной характеристики и сохранить остальные

#### Scenario: Очистка характеристики
- **WHEN** операция features выполняется в режиме `clear`
- **THEN** сервис ДОЛЖЕН очистить только выбранную характеристику уровня товара через `shopProduct` и не выполнять прямой широкий `DELETE` связей

#### Scenario: Защита характеристик SKU
- **WHEN** у товара есть строки выбранной характеристики с `sku_id IS NOT NULL`
- **THEN** сервис MUST NOT читать их как текущие значения товара, изменять или удалять их

#### Scenario: Защита других характеристик
- **WHEN** товар содержит значения других характеристик
- **THEN** операция MUST изменить только характеристику с подтверждённым `feature_id`, а все остальные характеристики MUST остаться без изменений

#### Scenario: Проверка принадлежности значений
- **WHEN** хотя бы один переданный value ID отсутствует или принадлежит другой характеристике
- **THEN** сервис MUST отклонить всю операцию до первой записи

#### Scenario: Дочерняя характеристика
- **WHEN** выбранная характеристика имеет `parent_id > 0`
- **THEN** сервис MUST отклонить её как не поддерживаемую текущим релизом

#### Scenario: Неизвестный формат характеристики
- **WHEN** формат типа характеристики не описан официальной документацией Shop-Script
- **THEN** сервис MUST отклонить операцию без fallback-записи в произвольную таблицу значений

#### Scenario: Батч-загрузка характеристик
- **WHEN** операция применяется к нескольким товарам
- **THEN** сервис MUST батчево загрузить допустимые значения и текущие связи уровня товара без N+1 запросов

#### Scenario: Категории
- **WHEN** операция categories добавляет, удаляет или заменяет основную категорию
- **THEN** сервис ДОЛЖЕН использовать подтверждённую category ID, поддерживать режимы `add`, `remove`, `replace_main` и не создавать новые категории в рамках массовой операции

## ADDED Requirements

### Requirement: Массовое редактирование видео
Операция `video` MUST безопасно устанавливать одну ссылку всем выбранным товарам либо очищать существующую ссылку через документированный объект `shopProduct`.

#### Scenario: Установка ссылки
- **WHEN** администратор подтверждает режим `set` с абсолютным HTTP(S) URL
- **THEN** сервис MUST проверить права на каждый товар, записать `video_url` через `shopProduct` и вызвать `save()`

#### Scenario: Очистка ссылки
- **WHEN** администратор подтверждает режим `clear`
- **THEN** сервис MUST сохранить пустой `video_url` для каждого выбранного товара через `shopProduct`

#### Scenario: Неверная ссылка
- **WHEN** режим `set` содержит пустую, относительную или не-HTTP(S) ссылку
- **THEN** сервис MUST вернуть локализованную ошибку до первой записи

#### Scenario: Ошибка сохранения товара
- **WHEN** сохранение любого товара завершается исключением
- **THEN** сервис MUST выполнить `ROLLBACK`, не записывать успешный журнал и не продолжать операцию как успешную

#### Scenario: Подтверждение и журнал
- **WHEN** массовая операция видео успешно завершена
- **THEN** она MUST пройти общий confirm-flow и создать запись `video` в `shop_masseditor_log`
