## Context

В десктопной версии MassEditor чекбокс «Выбрать все на странице» расположен в `<thead>` таблицы товаров (Backend.html:343). На мобильных (≤1024px) весь `thead` скрыт через CSS (`display: none`, masseditor.css:1266), потому что таблица трансформируется в card-layout. В результате на мобильном устройстве нет возможности одним действием отметить все товары на текущей странице.

Существующая кнопка «Выбрать все найденные товары» (`data-role="select-filter"`, Backend.html:334) видна и на мобильных, но она включает filter-режим (все товары по фильтру, а не только текущая страница) — это другой сценарий использования.

JS-логика выбора в `masseditor.js` единая для desktop и mobile. Десктопный select-all использует `data-role="select-all"` (строки 127, 997-1006).

## Goals / Non-Goals

**Goals:**
- Дать мобильному пользователю элемент «Выбрать все на странице» с тем же поведением, что и десктопный чекбокс в `thead`.
- Разместить элемент в зоне `.masseditor-table-card__actions` рядом с кнопкой «Выбрать все найденные товары».
- Элемент должен быть видим только на мобильных (≤1024px), на десктопе — скрыт.
- Переиспользовать существующую JS-логику `select-all` без дублирования.
- Сохранить текущий мобильный card-based дизайн без визуальных regressions.

**Non-Goals:**
- Изменение поведения «Выбрать все найденные товары» (filter-режим).
- Добавление новых backend API или контроллеров.
- Изменение десктопного layout.

## Decisions

### 1. Тип элемента: чекбокс с лейблом vs кнопка

**Решение:** Чекбокс `<input type="checkbox">` с `<label>` — визуально аналогичен десктопному чекбоксу в `thead`, обеспечивает единообразие UX и корректно отражает состояние (checked/unchecked/indeterminate).

**Альтернатива:** Кнопка-тоггл — менее интуитивна для «выбрать все / снять все», сложнее показать промежуточное состояние.

### 2. Размещение в DOM

**Решение:** Добавить новый элемент внутрь `.masseditor-table-card__actions` (Backend.html:332), рядом с кнопкой `select-filter`. На десктопе элемент скрыт через CSS, на мобильных — видим.

**Альтернатива:** Вынести в отдельный блок над таблицей — создает лишний DOM и может сломать мобильный layout.

### 3. JS-подключение

**Решение:** Использовать отдельный `data-role="select-all-page-mobile"` для нового чекбокса и подключить его к той же логике, что и десктопный `select-all`. В `masseditor.js` добавить получение элемента по селектору и обработчик, идентичный существующему (строки 997-1006). Функция `updateSelectionState` будет обновлять оба элемента синхронно.

**Альтернатива:** Использовать тот же `data-role="select-all"` — потребует `querySelectorAll` и цикла, сложнее различать элементы в обработчиках.

### 4. i18n

**Решение:** Переиспользовать существующий ключ `select_all_page` (i18nService:98) для `aria-label` нового чекбокса. Текущие переводы:
- `ru_RU`: «Отметить все товары на странице»
- `en_US`: «Select all products on this page»

Видимый текст рядом с чекбоксом будет задан через `<label>` с тем же ключом.

## Risks / Trade-offs

- **[Риск]** Indeterminate-состояние на чекбоксе может вести себя по-разному в мобильных браузерах → **Митигация:** Проверить на iOS Safari и Chrome Android; при необходимости заменить на CSS-стилизацию.
- **[Риск]** Двойной select-all (desktop + mobile) может рассинхронизироваться → **Митигация:** `updateSelectionState()` уже обновляет `selectAll.checked` — добавить туда же обновление мобильного элемента.
- **[Trade-off]** Добавление нового DOM-элемента незначительно увеличивает HTML → Приемлемо, т.к. элемент простой (checkbox + label).
