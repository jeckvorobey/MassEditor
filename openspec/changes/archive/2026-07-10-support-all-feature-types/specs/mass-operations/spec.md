## MODIFIED Requirements

### Requirement: Остатки, характеристики и категории
Операции остатков, характеристик и категорий ДОЛЖНЫ соблюдать текущие ограничения релиза.

#### Scenario: Остатки без выбранного склада
- **WHEN** операция stock не указывает склад
- **ТОГДА** сервис МОЖЕТ менять общий SKU count только для товаров без складского учета и ДОЛЖЕН отклонить товары со складскими остатками

#### Scenario: Характеристики всех типов
- **WHEN** операция features устанавливает значение характеристики
- **ТОГДА** сервис ДОЛЖЕН поддерживать все типы характеристик Shop-Script: `varchar`, `text`, `double`, `int`, `boolean`, `dimension.*`, `color`, `select`, `radio`, `range`, кроме множественных (`multiple=1`) и дочерних (`parent_id > 0`)

#### Scenario: Boolean характеристика
- **WHEN** операция features устанавливает значение для характеристики типа `boolean`
- **ТОГДА** сервис ДОЛЖЕН принимать значения `"1"` (да) или `"0"` (нет), сохранять в `shop_feature_values_varchar` и отклонять другие значения

#### Scenario: Dimension характеристика
- **WHEN** операция features устанавливает значение для характеристики типа `dimension.*` (length, weight, volume, area)
- **ТОГДА** сервис ДОЛЖЕН принимать числовое значение, сохранять в `shop_feature_values_double` и валидировать как numeric

#### Scenario: Color характеристика
- **WHEN** операция features устанавливает значение для характеристики типа `color`
- **ТОГДА** сервис ДОЛЖЕН принимать строковое значение (hex или название цвета), сохранять в `shop_feature_values_varchar` и не допускать пустое значение

#### Scenario: Select/Radio характеристика
- **WHEN** операция features устанавливает значение для характеристики типа `select` или `radio`
- **ТОГДА** сервис ДОЛЖЕН принимать значение из предопределённого списка, сохранять в `shop_feature_values_varchar` и отклонять значения, отсутствующие в списке

#### Scenario: Range характеристика
- **WHEN** операция features устанавливает значение для характеристики типа `range`
- **ТОГДА** сервис ДОЛЖЕН принимать числовое значение, сохранять в `shop_feature_values_double` и валидировать как numeric

#### Scenario: Неизвестный тип характеристики
- **WHEN** характеристика имеет тип, отсутствующий в конфигурации типов плагина
- **ТОГДА** сервис ДОЛЖЕН использовать fallback-логику: сохранять как varchar, логировать предупреждение

#### Scenario: Очистка характеристики любого типа
- **WHEN** операция features выполняется в режиме `clear` для характеристики любого типа
- **ТОГДА** сервис ДОЛЖЕН удалить запись из `shop_product_features` независимо от типа характеристики

#### Scenario: Категории
- **WHEN** операция categories добавляет, удаляет или заменяет основную категорию
- **ТОГДА** сервис ДОЛЖЕН использовать подтверждённую category ID, поддерживать режимы `add`, `remove`, `replace_main` и не создавать новые категории в рамках массовой операции

## ADDED Requirements

### Requirement: Типо-зависимый UI ввода значений
Backend MUST показывать подходящий элемент ввода для значения характеристики в зависимости от её типа.

#### Scenario: UI для строковых типов
- **WHEN** пользователь выбирает характеристику типа `varchar` или `color`
- **ТОГДА** UI ДОЛЖЕН показать текстовое поле `<input type="text">`

#### Scenario: UI для числовых типов
- **WHEN** пользователь выбирает характеристику типа `double`, `int`, `dimension.*` или `range`
- **ТОГДА** UI ДОЛЖЕН показать числовое поле `<input type="number" step="any">`

#### Scenario: UI для текстового типа
- **WHEN** пользователь выбирает характеристику типа `text`
- **ТОГДА** UI ДОЛЖЕН показать `<textarea>`

#### Scenario: UI для boolean типа
- **WHEN** пользователь выбирает характеристику типа `boolean`
- **ТОГДА** UI ДОЛЖЕН показать выпадающий список с вариантами «Да» (1) и «Нет» (0)

#### Scenario: UI для selectable типов
- **WHEN** пользователь выбирает характеристику типа `select` или `radio` с флагом `selectable=1`
- **ТОГДА** UI ДОЛЖЕН показать выпадающий список с доступными значениями из БД

#### Scenario: Переключение виджета при смене характеристики
- **WHEN** пользователь меняет выбранную характеристику в выпадающем списке
- **ТОГДА** JS ДОЛЖЕН скрыть текущий виджет ввода и показать виджет, соответствующий типу новой характеристики

### Requirement: Доступные значения selectable-характеристик
Backend MUST предзагружать и передавать в шаблон доступные значения для selectable-характеристик.

#### Scenario: Предзагрузка значений
- **WHEN** backend action загружает список характеристик
- **ТОГДА** для каждой характеристики с `selectable=1` сервис ДОЛЖЕН загрузить значения из соответствующей таблицы `shop_feature_values_*` и передать в шаблон как JSON-объект

#### Scenario: Безопасность SQL
- **WHEN** загружаются доступные значения характеристики
- **ТОГДА** запрос ДОЛЖЕН использовать параметризованный SQL с `feature_id` как integer и whitelist таблицы значений

### Requirement: Локализация типов характеристик
Все новые пользовательские тексты MUST быть локализованы на `ru_RU` и `en_US`.

#### Scenario: Новые i18n-ключи
- **WHEN** реализуется типо-зависимый UI
- **ТОГДА** MUST быть добавлены ключи: `feature_type_boolean_yes`, `feature_type_boolean_no`, `feature_type_color_hint`, `feature_type_dimension_hint`, `feature_type_range_hint` в обе локали
