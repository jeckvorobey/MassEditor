## Context

Фильтр склада `stock_id` уже нормализуется и ограничивает список товаров в `shopMasseditorPluginProductSelectionService`. После этого `getPage()` вызывает `attachStockDetails()` без контекста фильтра: сервис батчево агрегирует остатки всех складов, а шаблон показывает общий `product.count` и кнопку `Склады`, если найден хотя бы один складской остаток.

Нужно разделить два режима отображения без изменения состава выборки: выбранный склад задает контекст остатка, а значение `0` сохраняет режим обзора всех складов. Решение должно работать одинаково для desktop- и mobile-разметки, не добавлять запрос на каждый товар и не вводить новые пользовательские тексты.

## Goals / Non-Goals

**Goals:**

- Передать нормализованный `stock_id` из `getPage()` в batch-подготовку остатков.
- В режиме выбранного склада агрегировать остаток только по этому `stock_id`, включая `0` и `∞`.
- Представлять выбранный склад как единственную деталь остатка и скрывать multi-warehouse popover в шаблоне.
- В режиме `stock_id = 0` сохранить общий остаток и детали всех складов.
- Покрыть оба режима PHP-тестами и проверить шаблонную условность статическим тестом/проверкой.

**Non-Goals:**

- Не изменять SQL-условие фильтра товаров и выбор всех товаров по фильтру.
- Не менять массовую операцию `stock` или ее selector.
- Не менять схему БД, API и локализационные ключи.

## Decisions

1. `attachStockDetails()` получает второй аргумент `stock_id`, а `getPage()` передает туда `$filters['stock_id']`.

   Это оставляет reusable-логику в сервисе и не перегружает backend action. Значение уже прошло нормализацию и проверку доступности склада.

2. В выбранном складе batch-запрос группирует данные по `product_id` только для `ps.stock_id = i:selected_stock_id`, после чего сервис устанавливает отображаемый `product['count']` равным агрегату выбранного склада и формирует одну запись `stock_details`.

   Это исключает показ общего Shop-Script aggregate, который может включать другие склады. Альтернатива — вычислять значение в Smarty из массива всех складов — хуже из-за передачи лишних данных и риска расхождения с серверным агрегатом.

3. В режиме `stock_id = 0` сохраняется текущая ветка: детали строятся по всем настроенным складам, а `product['count']` остается исходным общим значением.

4. Шаблон проверяет `filters.stock_id` перед рендерингом кнопки `Склады` в обеих ячейках остатка. При выбранном складе `stock_details` остается данными для сервера/тестов, но popover не выводится.

5. Для бесконечного остатка используется существующий контракт: если хотя бы один SKU выбранного склада имеет `count IS NULL`, агрегат равен `null`, а UI показывает `∞`. При отсутствии строки выбранного склада значение равно `0` только для товаров, прошедших фильтр складской записи.

## Risks / Trade-offs

- [Risk] Общий `product.count` может быть использован другим участком страницы → Mitigation: менять его только в копии продукта, возвращаемой `getPage()`, и покрыть тестом оба режима.
- [Risk] Desktop и mobile шаблоны могут разойтись → Mitigation: обновить обе ветки и проверить отсутствие кнопки при `stock_id > 0`.
- [Risk] Дополнительная фильтрация batch SQL может создать N+1 → Mitigation: один запрос по всем `product_id` страницы с placeholder для выбранного склада.
- [Risk] Некорректный склад попадет в сервис напрямую → Mitigation: публичный `getPage()` всегда использует `normalizeFilters()`, а SQL применяет только приведенный integer `stock_id`.

## Migration Plan

Миграция БД не требуется. Развертывание состоит из обновления PHP-сервиса, шаблона и тестов. Откат — возврат вызова `attachStockDetails()` и условий шаблона к текущему поведению; URL и сохраненные фильтры остаются совместимыми.
