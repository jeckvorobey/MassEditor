## Why

Текущий OpenSpec-процесс фиксирует TDD, документацию Webasyst и общие проверки, но не задаёт единый обязательный маршрут по специализированным навыкам на этапах проработки, планирования, реализации и завершения. Из-за этого подходящие навыки могут применяться непоследовательно, а финальная проверка сложности и безопасности — пропускаться.

## What Changes

- Добавить repo-local навык `webasyst-loop`, который оркестрирует полный цикл разработки приложений, плагинов, виджетов и тем Webasyst/Shop-Script.
- Обязать OpenSpec загружать `webasyst-loop` для каждого нового функционального изменения и выбирать через него профильные навыки по типу продукта и этапу работы.
- Зафиксировать последовательность: исследование и документация → архитектура → OpenSpec-план → TDD-реализация → завершение → финальный review-loop.
- Сделать `complexity-optimizer` и `secure-review-loop` обязательными финальными проверками изменённого diff; замечания должны исправляться с повторным запуском проверок.
- Сохранить scope: навык не разрешает автоматически расширять задачу, архивировать change, коммитить, отправлять или публиковать изменения без соответствующего запроса пользователя.

## Capabilities

### New Capabilities

Нет.

### Modified Capabilities

- `quality-workflow`: полный цикл Webasyst-разработки должен выполняться через `webasyst-loop` с обязательным выбором профильных навыков и финальными проверками сложности и безопасности.

## Impact

- Новый навык: `.codex/skills/webasyst-loop/`.
- Точечное исключение в `.gitignore`, позволяющее поставлять `webasyst-loop` вместе с репозиторием без публикации остальных локальных `.codex`-материалов.
- Постоянный OpenSpec-контекст: `openspec/config.yaml`.
- Delta-спецификация процесса: `openspec/changes/add-webasyst-loop-skill/specs/quality-workflow/spec.md`.
- Runtime-код плагина, API и схема БД не изменяются.
- Для самого workflow-изменения релевантны официальные материалы по разработке плагинов и структуре файлов Webasyst; конкретные runtime-изменения в будущих change должны выбирать более узкие официальные источники по hooks, settings, `waPlugin` и другим затронутым механизмам.
