## Context

В репозитории уже существуют специализированные навыки для архитектуры Shop-Script-плагинов, TDD, аудита документации, security review, Store compliance и OpenSpec-операций. Однако OpenSpec-контекст не определяет единый навык-оркестратор, который обязан подобрать их на каждом этапе и не разрешает считать изменение завершённым до повторяемой финальной проверки.

Изменение не затрагивает runtime плагина. Оно добавляет repo-local workflow skill и закрепляет его использование в OpenSpec-контексте и delta-спецификации `quality-workflow`.

## Goals / Non-Goals

**Goals:**

- Задать единый полный цикл для новых функций, исправлений, новых плагинов, приложений, виджетов и тем Webasyst/Shop-Script.
- Выбирать минимальный достаточный набор профильных навыков по типу задачи и этапу, а не запускать все навыки без разбора.
- Обязать проверять официальную документацию до архитектурных решений и повторно сверять реализацию с подтверждёнными Webasyst API.
- Обеспечить TDD-последовательность и выполнение OpenSpec-задач до состояния `all_done` либо явного blocker.
- Сделать `complexity-optimizer` и `secure-review-loop` обязательными финальными gates с исправлением подтверждённых замечаний и повторной проверкой.

**Non-Goals:**

- Не изменять код, API, БД или UI плагина MassEditor.
- Не заменять профильные навыки Webasyst, OpenSpec, TDD и review их краткими пересказами.
- Не запускать публикацию, commit, push, PR, merge, sync или archive без соответствующего запроса пользователя либо явно заданного scope.
- Не требовать Store-материалы и packaging для обычного внутреннего изменения, если оно не готовится к релизу.

## Decisions

### 1. Использовать один repo-local навык-оркестратор

Создать `.codex/skills/webasyst-loop/SKILL.md`. Его frontmatter должен срабатывать на полный цикл разработки Webasyst/Shop-Script и на OpenSpec-изменения в таких проектах. Навык не дублирует подробные инструкции вложенных skills, а определяет порядок, условия выбора и критерии перехода между этапами.

Поскольку `.codex/` в проекте по умолчанию игнорируется, добавить точечные negation-правила `.gitignore` только для `webasyst-loop`. Не открывать для Git остальные локальные skills и generated security marker.

Альтернатива — расширить каждый OpenSpec skill отдельными Webasyst-правилами. Она отклонена: это создаёт дублирование и усложняет синхронизацию одинаковых gates между `propose`, `apply`, `sync` и `archive`.

### 2. Закрепить применение на двух уровнях

В `openspec/config.yaml` добавить постоянное правило: перед созданием или применением change загрузить `webasyst-loop`, а в proposal/design/tasks явно фиксировать выбранные профильные skills и проверки. В delta spec `quality-workflow` добавить проверяемый контракт процесса.

Альтернатива — полагаться только на description нового skill. Она отклонена: автоматический trigger полезен, но OpenSpec-контекст должен явно сохранять проектное правило.

### 3. Разделить обязательные и условные навыки

Обязательная основа цикла: OpenSpec skill текущего действия, `webasyst-shop-script-plugin-architect` для архитектурных/runtime-изменений, `webasyst-plugin-docs-auditor` для подтверждения Webasyst API, TDD skill для затронутого стека, затем `complexity-optimizer` и `secure-review-loop`.

Условно подключать `legacy-gap-product-builder` для нового коммерческого продукта/MVP, `php-security-reviewer` для PHP/backend mutation, `webasyst-store-compliance-reviewer` для релиза/публикации, `webasyst-changelog-writer` и `webasyst-plugin-description-writer` для соответствующих материалов.

### 4. Выполнять финальную проверку как исправляющий цикл

После завершения реализации сначала запустить `complexity-optimizer` на текущем diff и связанных runtime-файлах, затем `secure-review-loop`. Подтверждённые замечания исправлять через тот же TDD-проход, после чего повторять узкие тесты и оба финальных gates. Если обязательный subskill или его проверка недоступны, не заявлять полное завершение: указать непроверенную область и blocker/residual risk.

### 5. Сохранять буквальный scope и границы доставки

`webasyst-loop` координирует разработку, но не расширяет разрешения. Он должен сохранять backend-only или иной заданный scope, не менять core Webasyst, не добавлять лишний storefront и не выполнять delivery/archive автоматически без запроса.

## Risks / Trade-offs

- [Риск] Оркестратор станет слишком длинным и начнёт дублировать дочерние skills. → [Снижение] Оставить в `SKILL.md` только routing, gates и stop conditions; подробности читать из выбранных skills.
- [Риск] `complexity-optimizer` частично повторно запускается внутри `secure-review-loop`. → [Снижение] Считать первый запуск самостоятельным performance gate, а повтор внутри secure loop — проверкой после security-контекста и возможных исправлений.
- [Риск] Часть security subskills может быть недоступна в конкретной сессии. → [Снижение] Запретить формулировку о полном покрытии и явно фиксировать непроверенные зоны.
- [Риск] Все условные skills будут запускаться для каждого маленького изменения. → [Снижение] Добавить таблицу маршрутизации и выбирать только навыки, соответствующие затронутому продукту, стеку и выпускному этапу.
- [Риск] OpenSpec change будет формально завершён, а runtime-проверка пропущена. → [Снижение] Требовать evidence по тестам, документационному аудиту, complexity и security до отметки финальной задачи.
