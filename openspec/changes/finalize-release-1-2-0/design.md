## Context

Плагин уже содержит реализованный runtime версии `1.2.0`, но публикационный архив был собран до окончательного объединения изменений и отличается от production-дерева в семи файлах. Пары переводов также рассинхронизированы: текущие `.mo` не воспроизводятся из `.po`. Скриншоты пользователь проверяет отдельно и в scope change не входят.

Официальные требования Webasyst Store подтверждают формат `tar.gz`, единственную корневую директорию с ID продукта, числовой `vendor`, трехчастную версию, защитные `.htaccess`, Webasyst 2 compatibility и соответствие описания фактической функциональности.

Маршрутизация навыков: `webasyst-loop` оркестрирует change; `openspec-propose` и `openspec-apply-change` ведут артефакты и задачи; `webasyst-store-compliance-reviewer` проверяет публикационный кандидат; `complexity-optimizer` и `secure-review-loop` завершают review.

## Goals / Non-Goals

**Goals:**

- доказуемо синхронизировать `.mo` с `.po` для `ru_RU` и `en_US`;
- привести `docs/release-1.2.0.md` к фактическим возможностям кода;
- получить архив `dist/masseditor-1.2.0.tar.gz` с нулевым diff относительно production-дерева;
- проверить структуру, metadata и отсутствие материалов разработки;
- сохранить воспроизводимое evidence команд и результатов.

**Non-Goals:**

- исправление найденных пользователем UI/runtime-ошибок;
- изменение PHP/JS-бизнес-логики, API, БД, hooks, actions или пользовательских текстов;
- изменение скриншотов, цены или Store-карточки;
- загрузка архива во внешний Webasyst Store, commit, push или archive OpenSpec change.

## Decisions

1. До реализации фиксируются два красных состояния: byte mismatch скомпилированных `.mo` и ненулевой `diff -qr` распакованного архива. Это заменяет неприменимый PHP/JS unit-test для механической релизной сборки.
2. Gettext-компиляция выполняется доверенным `msgfmt --check` внутри Docker-контейнера `wa-php`, потому что на host `msgfmt` отсутствует. Результат передается в временный файл на host и только после успеха заменяет отслеживаемый `.mo`.
3. Архив собирается из временной staging-директории с единственным корнем `masseditor/`. Сначала проверяется кандидат, затем он заменяет `dist/masseditor-1.2.0.tar.gz`.
4. Сравнение кандидата выполняется после распаковки через рекурсивный diff с production-деревом, поэтому скрытые `.htaccess` и бинарные `.mo` входят в доказательство равенства.
5. `docs/release-1.2.0.md` описывает только уже реализованные функции: фильтр типа складского учёта, множественные общие характеристики и массовое видео. Неподдерживаемые SKU-/дочерние характеристики, создание значений и загрузка видеофайлов не обещаются.
6. Изменения не создают запросов к БД, циклов обработки каталога или новых границ доверия; N+1 и алгоритмическая сложность неприменимы к production diff, но обязательные scanners и ручная проверка всё равно выполняются.

## Risks / Trade-offs

- [Сбой `msgfmt` или невалидный `.po`] → не заменять текущий `.mo` и остановить сборку.
- [Частично собранный архив] → проверять временный кандидат и заменять финальный файл только после всех структурных проверок.
- [Лишние файлы staging] → копировать только production-корень и проверять manifest плюс нулевой diff.
- [Документация обещает лишнюю функцию] → сверять release-документ с OpenSpec `release-1-2-0-video-multiple-features-updates` и текущими тестами.
- [Скриншоты не соответствуют финальному UI] → пользователь выполняет эту проверку отдельно; риск явно остается за пределами change.
