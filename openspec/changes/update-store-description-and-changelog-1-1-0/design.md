## Context

`docs/store-description-1_1_0.md` содержит русскую и английскую Store-ready версии описания релиза `1.1.0`: базовые характеристики, складские операции и мобильные сценарии. В Git также существует двуязычная пара `docs/CHANGELOG.md` и `docs/CHANGELOG.en.md`.

После этой записи в `1.1.0` были реализованы фильтрация и отображение остатков по выбранному складу, корректная запись складского остатка, поддержка дополнительных типов характеристик, мобильный выбор текущей страницы, сброс формы при смене операции, полная RU/EN локализация уведомлений и исправление пункта плагина в расширенном меню Shop-Script. Публикационные материалы должны описывать эти факты компактно и одинаково на двух языках.

## Goals / Non-Goals

**Goals:**

- Обновить двуязычное описание Webasyst Store по фактической реализации `1.1.0`.
- Восстановить постоянную синхронную пару changelog-файлов и добавить короткую запись `1.1.0`.
- Устранить противоречия в связанных релизных документах.
- Оставить HTML описания совместимым с редактором Webasyst Store.
- Сделать структуру пригодной для добавления следующих версий без изменения формата.

**Non-Goals:**

- Не менять PHP, JavaScript, Smarty, CSS, локали, конфигурацию или версию плагина.
- Не создавать новые скриншоты и не менять бинарные публикационные материалы.
- Не добавлять неподтвержденные системные требования, цену, trial, внешние интеграции или маркетинговые обещания.
- Не документировать внутренние рефакторинги, не влияющие на пользователя.

## Decisions

### 1. Хранить changelog в двух синхронных файлах

Русская история хранится в `docs/CHANGELOG.md`, английская — в `docs/CHANGELOG.en.md`. Для каждой версии должны совпадать номер, дата, набор смысловых пунктов и порядок фактов. Это выбрано вместо одного смешанного файла, потому что отдельные документы проще копировать в соответствующие языковые поля и проверять на смысловой дрейф.

Существующие удаленные в рабочем дереве файлы восстанавливаются и обновляются; третий changelog-файл не создается.

### 2. Использовать компактную запись версии без внутренних деталей

Запись `1.1.0 — 2026-07-10` содержит не более пяти пользовательских пунктов: складские функции, характеристики, категории и выбор товаров, улучшения интерфейса/локализации, исправления складской записи и меню. Имена классов, SQL-поля, тестовые методы и OpenSpec change names не включаются.

Публикуемое содержимое версии хранится внутри fenced-блока `html`, как Store-описание: заголовок версии оформляется через `<p><strong>`, изменения — через `<ul>` и `<li>`. Markdown-заголовок файла используется только для локальной навигации и не предназначен для вставки в Webasyst Store. Это позволяет копировать HTML без дополнительного преобразования, поскольку Markdown-разметка в поле changelog Webasyst не поддерживается.

Альтернатива с подробными секциями `Added/Improved/Compatibility/Notes` отклонена для Store update note: она повторяет релизный документ и хуже соответствует коротким примерам Webasyst. В будущем подробные записи допустимы только для крупных выпусков, если они остаются синхронными на двух языках.

### 3. Обновлять Store-описание точечно

Описание `1.0.0` служит базовым текстом для следующих версий. Существующие позиционирование, вводные абзацы, структура и пользовательские возможности не переписываются: очередная версия копирует предыдущее описание и дополняет его только подтвержденными пользовательскими возможностями. Точечно изменять разрешено фактически устаревшие ограничения и переносить в changelog технические детали навигации по списку, которые не должны входить в основное описание плагина.

В основном Store-описании не создаются разделы «История состава версии», «Развитие версии», `Version history` или `Version progression`. Пагинация, сохранение выбора между страницами и мобильный выбор текущей страницы также относятся к двуязычному changelog соответствующего релиза. Структура `RU summary`, `EN summary`, `RU HTML description`, `EN HTML description` сохраняется во всех версиях.

### 4. Ограничить HTML безопасным набором Webasyst Store

В HTML-блоках используются только `b`, `i`, `u`, `em`, `strong`, `br`, `a`, `img`, `p`, `ul`, `ol`, `li`, `table`, `tr`, `td`, `th`, `tbody`, `thead`, `tfoot`, `blockquote`, `pre`, `code`. Не используются inline styles, `<style>`, классы для форматирования, скрипты, iframe, заголовки `h1`–`h6`, `div` и `span`.

### 5. Проверять факты по реализации и архивным изменениям

Источниками фактов служат текущий runtime-код, тесты, `plugin.php`, архивные OpenSpec proposal/spec и Git-история от `1.0.0` до текущего `1.1.0`. `docs/release-1.1.0.md` и `docs/store-publication.md` обновляются теми же формулировками, чтобы не сохранять старый запрет на цвета, диапазоны и размерности.

## Risks / Trade-offs

- [RU и EN версии разойдутся по смыслу] → Проверять одинаковое количество и порядок пунктов, затем выполнять покомпонентную сверку фактов.
- [Описание пообещает неподдерживаемую функцию] → Каждый новый пункт подтверждать кодом, тестом или завершенным OpenSpec change; неизвестные требования не добавлять.
- [Changelog станет слишком подробным] → Ограничить запись `1.1.0` пятью компактными пунктами без технических деталей.
- [Останутся противоречащие документы] → Выполнить поиск по фразам `базовые характеристики`, `basic feature`, `цвета`, `ranges`, `dimensions` во всех затронутых документах.
- [Нарушится формат Webasyst Store] → Проверить HTML-блоки описания и changelog по одному списку разрешенных тегов и убедиться, что внутри публикуемого changelog нет Markdown-списков.

## Migration Plan

1. Обновить Store-описание и два связанных релизных документа.
2. Восстановить синхронную пару changelog-файлов с записью `1.1.0`.
3. Проверить RU/EN эквивалентность, HTML-теги и отсутствие устаревших ограничений.
4. Выполнить OpenSpec-валидацию. Откат не требует миграции данных и сводится к возврату документационных файлов.
