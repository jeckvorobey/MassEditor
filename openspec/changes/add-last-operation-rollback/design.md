## Context

`shop_masseditor_log` хранит только тип, количество и текстовое описание операции; восстановить прежние значения из него невозможно. Поддерживаемые операции затрагивают поля `shop_product`, модификации, складские строки, теги, категории и одну характеристику. Локальный Shop-Script использует MyISAM для этих стандартных таблиц, поэтому существующие `START TRANSACTION`/`ROLLBACK` не дают достаточной гарантии от частичного результата: новая функция должна иметь собственный компенсирующий снимок и уметь восстановить исходное состояние при сбое фиксации или отката.

Официальная документация подтверждает POST/CSRF-поток Webasyst, плейсхолдеры `waModel`, метаобновления и записываемые свойства `skus`, `tags`, `features`, `categories` объекта `shopProduct`, его `save()` и `checkRights()`. Локальная схема Shop-Script подтверждена через `SHOW CREATE TABLE`; динамические имена таблиц значений характеристик уже ограничены существующим whitelist.

## Goals / Non-Goals

**Goals:**

- отменять только глобально последнюю успешную массовую операцию, если её выполнил текущий пользователь, прошло не более 3 часов и она ещё не отменена;
- сохранять точное состояние только изменяемых этой операцией полей до и после применения;
- перед откатом проверять права, CSRF, явное подтверждение, целостность снимка, актуальность записи и совпадение текущего состояния с зафиксированным результатом;
- восстанавливать данные ограниченными батчами и компенсировать частичный откат при исключении;
- не показывать снимки в журнале/UI/JSON, но писать отдельную запись аудита об успешной отмене;
- дать журналу локализованные кнопку, modal подтверждения, loader и безопасный результат.

**Non-Goals:**

- цепочка undo/redo, откат произвольной или более старой записи;
- восстановление изменений, сделанных вне исходной массовой операции;
- изменение состава массовых операций, фильтров, настроек или storefront;
- автоматический откат без явного действия пользователя;
- упаковка, публикация, commit/push или архивирование OpenSpec change.

## Decisions

### 1. Отдельный закрытый журнал снимков

Добавляются собственные таблицы `shop_masseditor_rollback` (заголовок) и `shop_masseditor_rollback_item` (состояние одного товара). Заголовок связывается с `shop_masseditor_log.id`, хранит пользователя, тип, срок и отметку отката; item хранит JSON `before_state`/`after_state`.

Разделение по товарам выбрано вместо одного большого JSON в логе: описание товара может быть крупным, а операция — включать до 1000 товаров. Item-строки записываются и читаются батчами, снимки не попадают в пользовательский аудит. Альтернатива — расширить `shop_masseditor_log`; отклонена из-за смешения отображаемого аудита и чувствительных данных восстановления.

### 2. Whitelist-снимок по типу операции

Сервис `shopMasseditorPluginRollbackService` получает нормализованный request и product IDs, батчево читает только разрешенные поля:

- `visibility`, `description`, `url`, `video`: соответствующее поле товара и `edit_datetime`;
- `price`, `compare_price`, `availability`, stock без склада: соответствующие поля SKU и затронутые агрегаты товара;
- stock выбранного склада: существование/значение строк всех складов, которые операция может создать или заменить, и агрегаты товара;
- `tags`: текущие имена тегов;
- `categories`: основная категория и полный набор связей категорий;
- `features`: только product-level связи подтвержденной характеристики; SKU-строки и другие характеристики не входят в снимок.

JSON проходит версионированную серверную валидацию; имена таблиц/полей никогда не принимаются из request или снимка. Альтернатива — полный дамп строк товара; отклонена, потому что он мог бы перезаписать несвязанные изменения.

### 3. Before/after и оптимистическая защита от перезаписи

До применения сохраняется `before_state`, после применения повторно читается `after_state`. Перед отменой текущее состояние тех же whitelisted полей ДОЛЖНО совпасть с `after_state`; иначе откат отклоняется как конфликт, чтобы не затереть более позднее ручное изменение Shop-Script.

Массовое применение и отмена сериализуются общей именованной DB-блокировкой на время проверки и записи. Это закрывает гонку между двумя запросами MassEditor. Внешние изменения Shop-Script не используют эту блокировку, поэтому дополнительной защитой остаётся сравнение текущего состояния непосредственно перед восстановлением.

### 4. Компенсирующее восстановление для MyISAM

Если применение изменило товары, но запись лога/снимка завершилась ошибкой, сервис восстанавливает `before_state`, удаляет неполный служебный аудит и пробрасывает исключение. Если откат падает после части записей, сервис восстанавливает `after_state`, оставляет исходную запись доступной только если компенсация завершилась, и пишет техническую ошибку только в `shop/plugins/masseditor.log`.

Для свойств, официально поддерживаемых `shopProduct`, восстановление использует `checkRights()` и `save()`. Прямой SQL применяется только к уже используемым проектом точечным полям/связям с integer casts и placeholders. После остатков выполняется штатный пересчёт агрегатов Shop-Script.

### 5. Eligibility и audit

Кнопка доступна, только если одновременно:

- `log_id` — максимальный ID в `shop_masseditor_log`;
- снимок принадлежит текущему пользователю;
- `expires_at >= now`, `rolled_back_at IS NULL`;
- action type совпадает в логе и снимке.

Controller принимает только POST, положительный integer `log_id` и `confirm_rollback=1`. После успешного восстановления заголовок атомарно помечается отменённым, затем создаётся log action `rollback` с ссылкой на исходный ID без старых значений. Следующая массовая операция автоматически делает предыдущую не последней.

### 6. UI и i18n

Backend action помечает `can_rollback` только одну допустимую строку текущей страницы и передаёт `rollback_url`. Smarty выводит отдельную CSRF-форму и экранированный `log_id`. Dependency-free JS инициализирует rollback до раннего выхода для вкладок без workspace form: кнопка открывает modal, подтверждение блокирует повторный запрос и включает существующий indeterminate loader-паттерн, результат выводится через `textContent`.

Новые ключи определяются в `shopMasseditorPluginI18nService`, включаются в JS dictionary и переводятся в `ru_RU`/`en_US` `.po`; PHP/JS-тесты проверяют выбранный `interface_language` и отсутствие русского fallback в JS.

## Risks / Trade-offs

- [Ручное изменение затронутого поля после операции] → сравнение с `after_state` отклоняет откат без записи.
- [MyISAM не поддерживает атомарную транзакцию стандартных таблиц] → before/after снимок, сериализация MassEditor и компенсирующее восстановление; сбой самой компенсации остаётся техническим blocker и логируется.
- [Крупные описания и до 1000 товаров] → отдельные item-строки, батчи, лимит числа/размера item и отсутствие N+1 чтений.
- [Повреждённый или подменённый JSON] → версия схемы, whitelist action/fields, точное соответствие product ID заголовку и отказ до записи.
- [Повторный клик/два администратора] → блокировка запроса, повторная серверная eligibility-проверка и условная отметка `rolled_back_at IS NULL`.
- [Утечка старого контента] → снимки не выбираются для UI и не включаются в response/log description; ошибки пользователю локализованы и не содержат SQL/JSON.
- [Старые установки не получают новые таблицы из `db.php`] → версия плагина повышается и добавляется идемпотентное метаобновление Webasyst.

## Migration Plan

1. Добавить обе таблицы в `lib/config/db.php` для новых установок.
2. Добавить метаобновление следующей версии, которое идемпотентно создаёт отсутствующие таблицы/индексы; повысить версию плагина.
3. Развернуть сервис/модель/controller/UI после схемы в одном обновлении.
4. Старые записи журнала не получают снимков и не показывают кнопку.
5. При откате версии кода таблицы можно оставить: они имеют префикс плагина и не влияют на старый runtime; удаление пользовательских снимков автоматически не выполняется.

## Open Questions

Нет. Контракт намеренно выбирает более безопасную трактовку: только создавший глобально последнюю операцию backend-пользователь может отменить её.
