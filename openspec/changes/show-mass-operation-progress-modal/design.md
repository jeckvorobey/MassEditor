## Context

Форма `Backend.html` сейчас выполняет обычный POST в `shopMasseditorPluginBackendAction`. После подтверждения пользователь видит прежнюю модалку до начала навигации, затем ждёт полный HTML-ответ без явного состояния. Серверная операция уже безопасно проверяет администратора, подтверждение, выбор, лимит и данные, выполняет транзакцию и журналирование.

Официальная документация Webasyst подтверждает backend route `?plugin=masseditor&action=apply`, класс `waJsonController`, поля `$response`/`$errors` и обязательное сохранение CSRF-защиты POST. Реализация использует `openspec`, `webasyst-development`, JS/PHP TDD, затем обязательные complexity/security gates.

## Goals / Non-Goals

**Goals:**

- Показать непрерывную обратную связь от отправки до итогового ответа.
- Не позволить закрыть или повторно отправить операцию во время выполнения.
- Показать локализованный результат и ждать ручного закрытия.
- После успешного закрытия обновить серверные данные страницы и дефолтную форму; после ошибки сохранить форму.
- Сохранить одинаковый request contract у обычного POST и JSON route.

**Non-Goals:**

- Не вводить фоновые очереди, polling, WebSocket/SSE или вычисляемый процент по товарам.
- Не менять бизнес-логику массовых операций, лимиты, транзакции, SQL и журнал.
- Не менять фильтры каталога или внешний интерфейс Store.

## Decisions

- Добавить `shopMasseditorPluginBackendApplyController extends waJsonController`. Официальный Webasyst JSON envelope (`status`, `data`, `errors`) будет единственным AJAX-контрактом.
- `waLongActionController` с polling рассмотрен как штатный механизм реального процента, но не используется в этом изменении: его `step()` выполняются разными HTTP-процессами, тогда как текущий сервис гарантирует одну общую транзакцию и rollback всей выборки. Переход потребует отдельного redesign с компенсацией частично завершённых шагов.
- Вынести чтение POST в `shopMasseditorPluginOperationRequestService`, чтобы HTML fallback и JSON controller использовали одинаковые ключи и типизацию без расхождения.
- JavaScript перехватит только подтверждённый submit при наличии `fetch` и `FormData`. Без этих API сохранится обычный POST fallback.
- Модалка выполнения будет отдельной от подтверждения. Во время запроса она имеет `aria-busy=true`, неопределённую progressbar-анимацию и недоступное закрытие. После ответа индикатор скрывается, появляется успех/ошибка и кнопка ручного закрытия.
- После успешного закрытия очистить сохранённый выбор и использовать `window.location.reload()`: это одновременно обновляет значения таблицы, последнюю операцию и серверные дефолты формы. После ошибки закрывается только result modal, а форма и выбор сохраняются.
- Новые строки идут из `shopMasseditorPluginI18nService` в `getJsTexts()`, `.po/.mo` для `ru_RU` и `en_US`, затем в шаблон/JS. Ответы сервера остаются локализованными по `interface_language`.

## Risks / Trade-offs

- [Нет точного процента выполнения] → показывать честный неопределённый progressbar без выдуманных процентов.
- [Повторный submit] → блокировать confirm submit и хранить флаг активного request до ответа.
- [Сетевая ошибка или невалидный JSON] → показывать локализованную общую ошибку, сохранять форму и разрешать ручное закрытие.
- [Ослабление защиты новым route] → повторить admin check, передавать CSRF из исходной формы, использовать тот же MassOperationService и скрывать неожиданные исключения с записью в plugin log.
- [N+1/сложность] → controller не добавляет чтение товаров; все батчи остаются внутри существующего сервиса.

## Migration Plan

Миграция БД не нужна. Откат состоит в удалении JSON route/result modal и возврате обычного POST; серверный fallback остаётся работоспособным на всём протяжении изменения.
