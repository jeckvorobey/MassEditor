## Context

Плагин `masseditor` работает только в backend Shop-Script и уже имеет общий безопасный поток: выбор товаров, подтверждение, серверная нормализация, транзакция и журнал. Текущая операция характеристик вручную удаляет и вставляет строки `shop_product_features`, не ограничивает удаление `sku_id IS NULL` и отвергает `multiple=1`; видео присутствует только как выключенный пункт UI; `lib/updates/` отсутствует.

Официальная документация задаёт три опорных контракта:

- `shopProduct` предназначен для изменения и сохранения товара, поле `video_url` и свойство `features` доступны для записи, изменения фиксируются вызовом `save()`;
- множественные значения характеристики передаются массивом;
- meta-updates плагина размещаются в `lib/updates/`, называются UNIX timestamp, должны быть повторно исполняемыми и последовательно учитываются Webasyst через `wa_app_settings.update_time`.

## Goals / Non-Goals

**Goals:**

- выпустить минимальный релиз `1.2.0` с тремя заявленными функциями;
- исключить ручное широкое удаление связей характеристик и делегировать сохранение документированному API товара;
- сохранить подтверждение, права, CSRF, транзакцию, журнал и локализацию;
- гарантировать точечную работу только с выбранной характеристикой уровня товара;
- использовать только штатные meta-updates Webasyst.

**Non-Goals:**

- видеофайлы, разные URL для каждого выбранного товара и управление эскизами;
- SKU-характеристики, дочерние характеристики и генерация модификаций;
- создание новых справочных значений в операции множественной характеристики;
- отдельная таблица миграций или перенос данных с отменённого идентификатора плагина;
- изменения storefront, marketplace API и экспорт журнала.

## Decisions

### 1. Один релиз и один OpenSpec change

Версия `1.2.0` объединяет три небольших взаимосвязанных изменения и их релизную проверку. Порядок реализации: безопасный контракт характеристик, видео, meta-update, затем интеграционный UI и релизная проверка. Это позволяет сначала убрать наиболее рискованную ручную запись данных.

Альтернатива — три отдельных релиза — отклонена, потому что увеличивает объём упаковки и ручной публикации без продуктовой пользы.

### 2. Видео сохраняется только через `shopProduct`

Сервис нормализует режим `set|clear`, для `set` проверяет непустой абсолютный HTTP(S) URL, затем создаёт `shopProduct`, проверяет право `checkRights()`, записывает `video_url` и вызывает `save()`. Формат и побочные действия хранения остаются ответственностью Shop-Script; прямой `UPDATE shop_product` и вызов недокументированных внутренних классов видео не используются.

Одна ссылка применяется ко всем выбранным товарам. Ошибка любого сохранения приводит к общему `ROLLBACK`, а успешная операция получает один журнал MassEditor.

### 3. Множественные значения работают только с существующими ID

UI передаёт массив `feature_value_ids[]`. До записи сервис:

- загружает выбранную корневую характеристику и проверяет `multiple=1`;
- удаляет дубликаты и приводит ID к положительным integer;
- одним батч-запросом проверяет, что все ID принадлежат выбранной характеристике;
- одним батч-запросом загружает текущие связи только уровня товара (`sku_id IS NULL`) для всех выбранных товаров;
- вычисляет итоговый набор для `replace`, `add`, `remove` или `clear`.

Итоговые значения передаются массивом в документированное записываемое свойство `features` объекта `shopProduct`, после чего вызывается `save()`. В запрос не передаются другие характеристики, а строки SKU не читаются и не меняются. Для одиночных характеристик сохраняется существующий режим `set|clear`, но запись также переводится на API товара, чтобы убрать опасное широкое удаление.

`add` объединяет текущий и выбранный наборы, `remove` вычитает только выбранные ID, `replace` полностью заменяет только выбранную характеристику, `clear` передаёт пустое значение только для неё. Порядок и дубликаты нормализуются до записи.

### 4. Actions остаются тонкими, подготовка данных батчевая

Backend action только читает поля формы и готовит view model. Вся серверная валидация и вычисление итоговых значений остаются в `shopMasseditorPluginMassOperationService` либо в выделенном небольшом helper/service, если основной класс станет хуже тестироваться. Списки значений для UI загружаются батчами, без запроса на каждую характеристику.

### 5. Только штатный meta-update Webasyst

Создаётся timestamp-файл внутри `lib/updates/1.2.0/`. Он идемпотентно проверяет схему `shop_masseditor_log` и добавляет только отсутствующие обязательные элементы. Новая установка по-прежнему получает полную схему из `lib/config/db.php`. Повторный запуск не меняет данные и не падает.

Собственная `shop_masseditor_migrations` не создаётся: она дублировала бы `wa_app_settings.update_time` и сама потребовала бы bootstrap-миграцию.

### 6. UI и локализация

Операция `video` становится доступной в медиагруппе. Для множественной характеристики UI показывает multi-select и только допустимые режимы. Подтверждение перечисляет операцию, режим и количество товаров, но не выводит неэкранированный ввод.

Источник текстов: ключи `shopMasseditorPluginI18nService` → данные `$texts`/`window.masseditorI18n` → Smarty/JS. Каждый новый ключ добавляется в `ru_RU` и `en_US`; тесты переключают `interface_language` и проверяют отсутствие русского fallback в английском режиме.

## Risks / Trade-offs

- [Различия форматов типов характеристик] → использовать документированные массивы значений и сначала покрыть selectable/multiple; сложные значения проверять на реальном bootstrap до релиза.
- [Частичное сохранение при исключении] → единая транзакция и обязательный `ROLLBACK`.
- [Повреждение SKU-характеристик] → читать только `sku_id IS NULL`, не выполнять прямой DELETE и проверять сохранность SKU-ряда тестом.
- [Значение от другой характеристики] → строгая проверка принадлежности всех ID до первой записи.
- [N+1 на подготовке] → батч-загрузка значений и текущих связей; запись выполняется батчами текущего сервиса.
- [Неподдерживаемая видеоссылка] → синтаксическая проверка HTTP(S), затем документированный `shopProduct::save()`; внутренние недокументированные API не вызываются.
- [Повторное meta-update] → проверка наличия каждого элемента схемы и тест двойного запуска.
- [Расхождение локалей] → одинаковый набор i18n-ключей и PHP/JS-тесты обеих локалей.

## Migration Plan

1. Установить обновление `1.2.0` через штатный Installer Webasyst.
2. Webasyst выполняет timestamp meta-update в хронологическом порядке и обновляет `update_time`.
3. Новые операции становятся доступны после обновления файлов и словарей.
4. При ошибке meta-update Webasyst не запускает последующие обновления; исправленный идемпотентный файл можно выполнить повторно.
5. Откат кода не откатывает пользовательские массовые изменения; для схемы destructive rollback не требуется, потому что meta-update ничего не удаляет.
