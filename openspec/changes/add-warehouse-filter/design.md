## Context

MassEditor уже загружает список складов через `shopMasseditorPluginProductSelectionService::getStocks()` и отображает остатки по складам батчем через `attachStockDetails()`. Фильтры списка товаров сейчас нормализуются в `shopMasseditorPluginProductSelectionService::normalizeFilters()` и применяются в `buildConditions()` только для `query`, `status`, `availability` и `category_id`.

Официальная документация Webasyst подтверждает, что backend-экшены плагина живут в `lib/actions`, reusable-код - в `lib/classes`, модели - в `lib/models`, а backend URL плагина использует параметр `plugin=PLUGIN`. Для SQL-части перед реализацией нужно опираться на существующий локальный паттерн `waModel::query()` с плейсхолдерами и приведением типов.

## Goals / Non-Goals

**Goals:**

- Добавить серверный фильтр `stock_id` в существующий pipeline выборки товаров.
- Сделать фильтр доступным в backend UI как `Склад` / `Warehouse`.
- Обеспечить одинаковое поведение для страницы товаров, подсказок поиска и выбора всех товаров по текущему фильтру.
- Сохранить безопасность SQL и батчевую загрузку связанных данных.

**Non-Goals:**

- Не менять схему БД.
- Не менять правила массовой операции `stock` и ее отдельный selector склада.
- Не добавлять фильтры по диапазону остатка или по режиму бесконечного остатка.
- Не менять storefront или внешние API.

## Decisions

1. Фильтр хранится как `stock_id` в общем объекте фильтров.

   Причина: это соответствует уже существующим `category_id` и параметрам URL, упрощает сохранение состояния в пагинации, POST payload для выбора по фильтру и подсказках. Альтернатива с отдельным именем вроде `warehouse_id` хуже, потому что в Shop-Script таблица и текущая операция уже используют термин `stock_id`.

2. Некорректный `stock_id` нормализуется в `0`.

   Причина: поведение должно быть безопасным и аналогичным неизвестным `status`/`availability`. При реализации нужно проверить ID по списку из `shop_stock`, чтобы удаленный или чужой склад не давал некорректный SQL-контракт.

3. Условие склада добавляется через `EXISTS`:

   ```sql
   EXISTS (
       SELECT 1
       FROM shop_product_skus ss
       INNER JOIN shop_product_stocks ps ON ps.sku_id = ss.id
       WHERE ss.product_id = p.id AND ps.stock_id = i:stock_id
   )
   ```

   Причина: `EXISTS` не размножает строки основной выборки, не требует новых `GROUP BY` полей и совместим с текущими `COUNT(DISTINCT p.id)` и `SELECT DISTINCT p.id`. Альтернатива через общий `INNER JOIN` проще визуально, но повышает риск дублирования строк при нескольких SKU.

4. Фильтр склада не обязан проверять ненулевой остаток.

   Причина: пользовательский запрос звучит как фильтр “Склад”, а не “есть остаток на складе”. В Shop-Script наличие записи в `shop_product_stocks` означает складской учет SKU на этом складе; фильтр должен показать товары, связанные с выбранным складом, включая нулевой остаток. Фильтр “положительный остаток” лучше выделять отдельным будущим требованием.

5. UI использует уже загруженный `$stocks`.

   Причина: action уже получает список складов для операции stock и отображения остатков. Повторно используем этот набор для select фильтра, не добавляя второй запрос или новый action.

## Risks / Trade-offs

- [Risk] У пользователя может быть ожидание “только товары с остатком > 0” → Mitigation: в тексте требований явно фиксируем, что фильтр выбирает товары, у которых есть складская запись/SKU на выбранном складе; диапазон остатка не входит в scope.
- [Risk] Подсказки поиска могут забыть новый фильтр → Mitigation: добавить тест и обновить `buildSearchSuggestionsUrl()` при необходимости.
- [Risk] Пагинация или вкладка лога потеряет `stock_id` → Mitigation: обновить URL-шаблоны всех мест, где уже сохраняются `query/status/availability/category_id`.
- [Risk] Складская фильтрация может дать N+1 при проверке ID склада → Mitigation: нормализовать через один батчевый список складов или один запрос к `shop_stock`, не проверять склад на каждый товар.

## Migration Plan

Миграция БД не требуется. Для отката достаточно удалить UI-control и вернуть нормализацию/SQL к прежним фильтрам; существующие URL без `stock_id` продолжают работать, потому что `0` означает “все склады”.
