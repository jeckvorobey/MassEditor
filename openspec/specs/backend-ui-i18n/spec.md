## Purpose

Зафиксировать контракт backend UI, Smarty-шаблона, JavaScript-поведения и локализации `ru_RU`/`en_US`.

## Requirements

### Requirement: Backend UI вкладки
Backend-экран MUST оставаться единым экраном плагина с вкладками `products`, `log`, `settings` и URL `?plugin=masseditor`.

#### Scenario: Неизвестная вкладка
- **WHEN** request содержит неизвестный `view`
- **THEN** backend action MUST открыть вкладку `products`

#### Scenario: Сохранение состояния
- **WHEN** пользователь переходит между вкладками или страницами
- **THEN** ссылки MUST сохранять текущие фильтры, включая `query`, `status`, `availability`, `category_id`, `stock_id`, страницу товаров и страницу журнала там, где это применимо

### Requirement: Smarty-шаблон и экранирование
Шаблон `templates/actions/backend/Backend.html` MUST выводить пользовательские и серверные данные с HTML/URL escaping и включать CSRF-токен для POST-форм.

#### Scenario: Вывод данных товара или ошибки
- **WHEN** шаблон показывает название, описание, категории, теги, ошибку или результат операции
- **THEN** значение MUST быть экранировано через Smarty `escape`

#### Scenario: POST-форма
- **WHEN** форма применяет операцию или сохраняет настройки
- **THEN** в форме MUST присутствовать `{$wa->csrf()}`

### Requirement: JavaScript без зависимостей
`js/masseditor.js` MUST оставаться простым dependency-free JavaScript, который управляет выбором товаров, localStorage, переключением операций, клиентской валидацией, confirm modal, toast-уведомлениями и подсказками поиска.

#### Scenario: Выбор товаров
- **WHEN** пользователь отмечает товары на странице
- **THEN** JS MUST сохранить ID в `localStorage` с ключом, привязанным к plugin ID, и восстановить выбор при перезагрузке

#### Scenario: Смена операции
- **WHEN** пользователь выбрал другую операцию в списке операций
- **THEN** text/textarea предыдущей формы ДОЛЖНЫ очиститься, select ДОЛЖНЫ вернуться к первому варианту, а выбор товаров и фильтры ДОЛЖНЫ сохраниться

#### Scenario: Повторный выбор операции
- **WHEN** пользователь повторно выбирает уже активную операцию
- **THEN** заполненные значения этой операции ДОЛЖНЫ сохраниться

#### Scenario: Подтверждение операции
- **WHEN** пользователь открывает confirm modal
- **THEN** JS MUST проверить обязательные поля текущей операции, заполнить summary и перед submit добавить hidden product IDs и `confirm_apply=1`

### Requirement: Локализация интерфейса
Видимый backend-текст ДОЛЖЕН (MUST) идти через `shopMasseditorPluginI18nService`, `.po`/`.mo` каталоги и JS i18n dictionary, без новых хардкодных русских строк в JS fallback. Для каждой поддерживаемой локали `ru_RU` и `en_US` словарь ДОЛЖЕН (MUST) содержать перевод всех используемых ключей, включая toast-заголовки, сообщения валидации, серверные уведомления и доступные подписи (`aria-label`). Английские fallback-строки в `masseditor.js` допустимы только как защита от отсутствующего ключа и не ДОЛЖНЫ (MUST NOT) отображаться при корректно сформированном русском словаре.

#### Scenario: Выбор языка
- **WHEN** настройка `interface_language` задана как `ru_RU` или `en_US`
- **THEN** backend UI, JS-тексты, toast-уведомления и доступные подписи ДОЛЖНЫ использовать выбранный язык независимо от локали Webasyst

#### Scenario: Настройка языка не задана
- **WHEN** `interface_language` пустая
- **THEN** язык ДОЛЖЕН определяться по `wa()->getLocale()`: английские локали открывают `en_US`, остальные `ru_RU`

#### Scenario: Ошибка выбора склада на русском
- **WHEN** при `interface_language=ru_RU` пользователь выбрал товар со складским учетом, оставил «Без склада» и пытается применить остаток
- **THEN** toast ДОЛЖЕН показать русские заголовок «Ошибка», текст «Для товаров со складским учетом выберите конкретный склад.» и доступную подпись «Закрыть уведомление»

#### Scenario: Ошибка выбора склада на английском
- **WHEN** при `interface_language=en_US` пользователь выбрал товар со складским учетом, оставил no warehouse и пытается применить остаток
- **THEN** toast ДОЛЖЕН показать английские заголовок, текст и доступную подпись из `en_US` словаря

#### Scenario: JS fallback
- **WHEN** `window.masseditorI18n` не содержит ключа
- **THEN** fallback-строка в `masseditor.js` ДОЛЖНА оставаться английской, а русские строки ДОЛЖНЫ приходить из i18n dictionary

#### Scenario: Фильтр склада
- **WHEN** backend UI показывает форму фильтра товаров
- **THEN** label `Склад` и вариант `Все склады` MUST приходить из i18n-слоя плагина и MUST иметь переводы `ru_RU` и `en_US`
- **AND** выбранный `stock_id` MUST восстанавливаться после submit, пагинации и возврата на вкладку товаров

### Requirement: Языковой контроль OpenSpec
Документация OpenSpec проекта ДОЛЖНА (MUST) требовать проверки полноты `ru_RU` и `en_US` для каждого нового или измененного пользовательского backend-текста, включая уведомления и сообщения валидации.

#### Scenario: Новое OpenSpec-изменение затрагивает видимый текст
- **WHEN** proposal, design или tasks описывают новый либо измененный видимый текст backend UI
- **THEN** автор ДОЛЖЕН указать оба языка, источник ключа i18n и тест, который проверяет отображение согласно `interface_language`

### Requirement: Читаемая типографика backend UI
Backend UI MassEditor MUST использовать согласованную типографическую шкалу, в которой основной текст, табличные данные, подписи элементов и интерактивные controls остаются комфортно читаемыми на desktop и mobile.

#### Scenario: Таблицы и панели на desktop
- **WHEN** пользователь открывает вкладку `products`, `log` или `settings` на desktop-экране
- **THEN** основной текст панелей и таблиц MUST быть визуально читаемым без опоры на `10–12px` как основной размер для значений и действий

#### Scenario: Мобильные карточки и controls
- **WHEN** пользователь открывает backend UI на mobile-ширине
- **THEN** текст в карточках товаров, логе, кнопках, input и select MUST быть увеличен относительно desktop microcopy и MUST оставаться удобным для чтения и нажатия

#### Scenario: Scope без изменения пользовательского потока
- **WHEN** внедряется улучшение типографики
- **THEN** текущая структура вкладок, таблиц, карточек, confirm modal и локализованных текстов MUST сохраняться без изменения backend-логики

### Requirement: UI видео и множественных характеристик
Backend UI MUST показывать только поля и режимы, допустимые для выбранной операции и типа характеристики, и MUST передавать серверу нормализуемые значения без обхода общего подтверждения.

#### Scenario: Форма видео
- **WHEN** пользователь выбирает операцию `video`
- **THEN** UI MUST показать режимы `set`/`clear`, поле URL только для `set` и локализованную клиентскую ошибку для неверного URL

#### Scenario: Форма множественной характеристики
- **WHEN** пользователь выбирает характеристику с `multiple=1`
- **THEN** UI MUST показать режимы `replace`, `add`, `remove`, `clear` и multi-select существующих значений этой характеристики

#### Scenario: Очистка без значений
- **WHEN** выбран режим `clear` для видео или характеристики
- **THEN** UI MUST скрыть и отключить поле значения, чтобы устаревший ввод не попал в POST

#### Scenario: Подтверждение
- **WHEN** обязательные поля валидны и пользователь открывает modal
- **THEN** modal MUST показать локализованные название операции, режим и число товаров, а submit MUST добавить `confirm_apply=1`

### Requirement: Локализация релиза 1.2.0
Все новые видимые строки видео и множественных характеристик MUST поступать из `shopMasseditorPluginI18nService` и JS i18n dictionary и MUST иметь значения в `ru_RU` и `en_US`.

#### Scenario: Русский интерфейс
- **WHEN** `interface_language=ru_RU`
- **THEN** подписи режимов, URL, multi-select, подтверждение и ошибки MUST отображаться по-русски

#### Scenario: Английский интерфейс
- **WHEN** `interface_language=en_US`
- **THEN** подписи режимов, URL, multi-select, подтверждение и ошибки MUST отображаться по-английски без русского fallback

#### Scenario: Экранирование
- **WHEN** UI отображает видеоссылку, название характеристики, значение или серверную ошибку
- **THEN** Smarty/JS MUST использовать безопасный текстовый вывод без интерпретации пользовательского HTML
