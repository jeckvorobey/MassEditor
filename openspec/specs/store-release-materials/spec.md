## Purpose

Зафиксировать требования к двуязычным описаниям, changelog и публикационным документам MassEditor для Webasyst Store.

## Requirements

### Requirement: Двуязычное описание Webasyst Store
Документ `docs/store-description-1_1_0.md` ДОЛЖЕН (MUST) содержать фактически эквивалентные русские и английские summary и HTML-описания, отражающие реализованные возможности версии плагина.

#### Scenario: Описание возможностей версии 1.1.0
- **WHEN** описание обновляется для версии `1.1.0`
- **THEN** обе языковые версии ДОЛЖНЫ перечислять массовое изменение остатков, складской фильтр и остаток выбранного склада, а также поддерживаемые типы характеристик
- **AND** обе версии ДОЛЖНЫ одинаково ограничивать SKU-, дочерние и множественные характеристики и создание новых значений

#### Scenario: Неподтвержденная возможность
- **WHEN** функция не подтверждена текущим кодом, тестами или завершенным OpenSpec change
- **THEN** описание НЕ ДОЛЖНО обещать эту функцию пользователю

### Requirement: Накопительное Store-описание без истории версий
Каждое следующее Store-описание ДОЛЖНО (MUST) сохранять позиционирование, вводные абзацы, структуру и ранее опубликованные пользовательские возможности из предыдущей версии, дополняя их только новыми подтвержденными данными. Устаревшее ограничение MAY быть точечно исправлено, если новая реализованная функция сделала его фактически неверным. Технические детали навигации по списку MUST храниться в changelog соответствующей версии, а не в основном описании плагина.

#### Scenario: Подготовка описания следующей версии
- **WHEN** создается Store-описание версии после `1.0.0`
- **THEN** RU- и EN-описания MUST наследовать текст предыдущей версии без переписывания существующих преимуществ
- **AND** новые возможности MUST добавляться в существующие тематические разделы
- **AND** основное описание MUST NOT содержать отдельный блок истории версий, включая «История состава версии», «Развитие версии», `Version history` и `Version progression`

#### Scenario: Пагинация и мобильный выбор страницы
- **WHEN** описание содержит сведения о пагинации, сохранении выбора между страницами или выборе текущей страницы на мобильных устройствах
- **THEN** эти сведения MUST быть удалены из основного Store-описания
- **AND** пагинация MUST быть отражена в changelog `1.0.0`, а мобильный выбор текущей страницы MUST быть отражен в changelog `1.1.0`

#### Scenario: Новая функция отменяет прежнее ограничение
- **WHEN** реализованная возможность делает отдельное ограничение предыдущего описания неверным
- **THEN** MUST изменяться только соответствующее ограничение и связанные с ним фактические данные
- **AND** остальные части предыдущего описания MUST сохраняться

### Requirement: Совместимое HTML-форматирование Store
HTML-блоки описания ДОЛЖНЫ (MUST) использовать только базовые теги, совместимые с редактором Webasyst Store, без активного содержимого и CSS-зависимого оформления.

#### Scenario: Проверка HTML описания
- **WHEN** RU и EN HTML-блоки готовы к публикации
- **THEN** они ДОЛЖНЫ содержать только разрешенные теги `b`, `i`, `u`, `em`, `strong`, `br`, `a`, `img`, `p`, `ul`, `ol`, `li`, `table`, `tr`, `td`, `th`, `tbody`, `thead`, `tfoot`, `blockquote`, `pre`, `code`
- **AND** они НЕ ДОЛЖНЫ содержать inline styles, `<style>`, скрипты, iframe, `h1`–`h6`, `div` или `span`

### Requirement: Синхронная двуязычная история версий
История выпусков ДОЛЖНА (MUST) храниться в `docs/CHANGELOG.md` для русского языка и `docs/CHANGELOG.en.md` для английского языка как синхронная пара с одинаковыми версиями, датами и фактическим составом записей. Публикуемое содержимое каждой версии ДОЛЖНО быть оформлено копируемым Store-ready HTML-блоком, а не Markdown-списком.

#### Scenario: Добавление новой версии
- **WHEN** готовится changelog очередной версии
- **THEN** запись ДОЛЖНА быть добавлена в оба файла в одном изменении
- **AND** RU и EN записи ДОЛЖНЫ иметь одинаковый порядок фактов без добавления возможностей только в одном языке

#### Scenario: Повторное обновление существующей версии
- **WHEN** текущая версия уже присутствует в changelog
- **THEN** существующий раздел ДОЛЖЕН быть обновлен без создания дубликата версии

#### Scenario: Формат для вставки в Webasyst Store
- **WHEN** запись версии подготовлена для публикации
- **THEN** каждый языковой файл ДОЛЖЕН содержать fenced-блок `html` с заголовком через `<p><strong>` и списком изменений через `<ul>` и `<li>`
- **AND** публикуемый блок НЕ ДОЛЖЕН содержать Markdown-маркеры списка

### Requirement: Компактная запись версии 1.1.0
Запись `1.1.0 — 2026-07-10` ДОЛЖНА (MUST) кратко описывать пользовательские изменения относительно `1.0.0` и быть пригодной для переноса в раздел обновлений Webasyst Store.

#### Scenario: Содержание записи 1.1.0
- **WHEN** формируется запись версии `1.1.0`
- **THEN** она ДОЛЖНА содержать не более пяти пунктов о складских возможностях, поддержке характеристик, категориях и выборе товаров, интерфейсе и локализации, исправлениях складской записи и пункта меню
- **AND** она НЕ ДОЛЖНА включать внутренние имена классов, SQL-поля, тестовые методы или OpenSpec change names

#### Scenario: Исправление пункта меню
- **WHEN** перечисляются исправления версии `1.1.0`
- **THEN** RU запись ДОЛЖНА сообщать об исправлении отображения пункта плагина в меню админ-панели Shop-Script, а EN запись ДОЛЖНА передавать тот же факт

### Requirement: Согласованность релизных документов
Документы `docs/release-1.1.0.md`, `docs/store-publication.md`, `docs/store-description-1_1_0.md` и оба changelog-файла ДОЛЖНЫ (MUST) согласованно описывать границы версии `1.1.0`.

#### Scenario: Проверка устаревших ограничений
- **WHEN** публикационные документы подготовлены
- **THEN** ни один из них НЕ ДОЛЖЕН утверждать, что цвета, диапазоны и размерности полностью не поддерживаются, если текущая реализация их поддерживает
- **AND** ограничения SKU-, дочерних и множественных характеристик ДОЛЖНЫ сохраняться одинаково во всех применимых документах
