## Purpose

Зафиксировать безопасный поток массовых операций над товарами, ограничения операций и обязательные серверные гарантии.

## Requirements

### Requirement: Безопасный поток применения
Массовая операция MUST проходить через выбор товаров, параметры операции, явное модальное подтверждение, серверную повторную валидацию, батчевое применение и запись в журнал.

#### Scenario: Нет подтверждения
- **WHEN** POST-запрос на применение операции не содержит `confirm_apply`
- **THEN** сервис MUST отклонить операцию до записи данных

#### Scenario: Недостаточно прав
- **WHEN** текущий пользователь не является администратором Shop-Script
- **THEN** backend action, JSON controller и сервис операции MUST отклонить запрос

#### Scenario: Слишком много товаров
- **WHEN** количество выбранных товаров превышает `operation_limit`
- **THEN** сервис MUST отклонить операцию и не начинать транзакцию

### Requirement: Транзакционность и логирование
Применение операции MUST выполняться в транзакции, а успешный результат MUST записываться в `shop_masseditor_log`.

#### Scenario: Успешное применение
- **WHEN** операция валидна и товары загружены
- **THEN** сервис MUST выполнить `START TRANSACTION`, применить изменения батчами, записать лог и выполнить `COMMIT`

#### Scenario: Ошибка во время применения
- **WHEN** во время записи возникает исключение
- **THEN** сервис MUST выполнить `ROLLBACK`, пробросить ошибку, а UI MUST скрыть технические детали от пользователя и записать их в `shop/plugins/masseditor.log`

### Requirement: Поддерживаемые операции
Сервис MUST поддерживать только whitelisted операции: `price`, `compare_price`, `visibility`, `availability`, `description`, `tags`, `url`, `stock`, `features`, `categories`.

#### Scenario: Неизвестная операция
- **WHEN** request содержит operation вне whitelist
- **THEN** сервис MUST вернуть локализованную ошибку неизвестной операции

#### Scenario: Цена и compare price
- **WHEN** выполняется операция цены или compare price
- **THEN** сервис MUST поддерживать режимы `set`, `add`, `subtract`, `increase_percent`, `decrease_percent`, округление 0-4 знака и запрет отрицательного результата

#### Scenario: Описание, теги и URL
- **WHEN** выполняются операции контента
- **THEN** сервис MUST валидировать непустые значения, поддерживать режимы описания `replace`/`prepend`/`append`, теги `add`/`remove`/`replace` и URL `regenerate`/`template`

### Requirement: Остатки, характеристики и категории
Операции остатков, характеристик и категорий MUST соблюдать текущие ограничения релиза.

#### Scenario: Остатки без выбранного склада
- **WHEN** операция stock не указывает склад или передает `stock_id = 0`
- **THEN** сервис МОЖЕТ менять общий SKU count только для товаров без складского учета и ДОЛЖЕН отклонить товары, у которых есть любые строки в `shop_product_stocks`

#### Scenario: Остатки выбранного склада
- **WHEN** операция stock передает `stock_id > 0`
- **THEN** сервис MUST записывать итоговый остаток только в `shop_product_stocks` для выбранного склада вместе с корректным `product_id` и MUST NOT напрямую сохранять новое значение в `shop_product_skus.count`

#### Scenario: Новый складской остаток товара
- **WHEN** операция stock передает `stock_id > 0`, а у SKU товара нет строки выбранного склада
- **THEN** сервис MUST считать старое значение выбранного склада равным `0`, MUST создать строку выбранного склада и MUST создать нулевые строки остальных настроенных складов для корректного пересчета общего остатка

#### Scenario: Режимы складского остатка
- **WHEN** операция stock передает `stock_id > 0`
- **THEN** сервис MUST поддерживать режимы `set`, `increase`, `decrease` и `infinite`, где `set` задает значение, `increase` прибавляет к старому значению выбранного склада, `decrease` запрещает отрицательный итог, а `infinite` записывает `NULL`

#### Scenario: Пересчет агрегатов после складского остатка
- **WHEN** операция stock успешно записывает остаток выбранного склада
- **THEN** сервис MUST обновить `shop_product.edit_datetime`, MUST пересчитать агрегаты товара через механизм Shop-Script и MUST отобразить общий числовой остаток вместо `∞`, если все складские остатки числовые

#### Scenario: Характеристики всех типов
- **WHEN** операция features устанавливает значение
- **THEN** сервис ДОЛЖЕН поддерживать все типы характеристик Shop-Script: `varchar`, `text`, `double`, `int`, `boolean`, `dimension.*`, `color`, `select`, `radio`, `range`, кроме множественных (`multiple=1`) и дочерних (`parent_id > 0`)

#### Scenario: Boolean характеристика
- **WHEN** операция features устанавливает значение для характеристики типа `boolean`
- **THEN** сервис ДОЛЖЕН принимать значения `"1"` (да) или `"0"` (нет), сохранять в `shop_feature_values_varchar` и отклонять другие значения

#### Scenario: Dimension характеристика
- **WHEN** операция features устанавливает значение для характеристики типа `dimension.*` (length, weight, volume, area)
- **THEN** сервис ДОЛЖЕН принимать числовое значение, сохранять в `shop_feature_values_double` и валидировать как numeric

#### Scenario: Color характеристика
- **WHEN** операция features устанавливает значение для характеристики типа `color`
- **THEN** сервис ДОЛЖЕН принимать строковое значение (hex или название цвета), сохранять в `shop_feature_values_varchar` и не допускать пустое значение

#### Scenario: Select/Radio характеристика
- **WHEN** операция features устанавливает значение для характеристики типа `select` или `radio`
- **THEN** сервис ДОЛЖЕН принимать значение из предопределённого списка, сохранять в `shop_feature_values_varchar` и отклонять значения, отсутствующие в списке

#### Scenario: Range характеристика
- **WHEN** операция features устанавливает значение для характеристики типа `range`
- **THEN** сервис ДОЛЖЕН принимать числовое значение, сохранять в `shop_feature_values_double` и валидировать как numeric

#### Scenario: Неизвестный тип характеристики
- **WHEN** характеристика имеет тип, отсутствующий в конфигурации типов плагина
- **THEN** сервис ДОЛЖЕН использовать fallback-логику: сохранять как varchar, логировать предупреждение

#### Scenario: Очистка характеристики любого типа
- **WHEN** операция features выполняется в режиме `clear` для характеристики любого типа
- **THEN** сервис ДОЛЖЕН удалить запись из `shop_product_features` независимо от типа характеристики

#### Scenario: Категории
- **WHEN** операция categories добавляет, удаляет или заменяет основную категорию
- **THEN** сервис ДОЛЖЕН использовать подтверждённую category ID, поддерживать режимы `add`, `remove`, `replace_main` и не создавать новые категории в рамках массовой операции

### Requirement: Типо-зависимый UI ввода значений
Backend MUST показывать подходящий элемент ввода для значения характеристики в зависимости от её типа.

#### Scenario: UI для строковых типов
- **WHEN** пользователь выбирает характеристику типа `varchar` или `color`
- **THEN** UI ДОЛЖЕН показать текстовое поле `<input type="text">`

#### Scenario: UI для числовых типов
- **WHEN** пользователь выбирает характеристику типа `double`, `int`, `dimension.*` или `range`
- **THEN** UI ДОЛЖЕН показать числовое поле `<input type="number" step="any">`

#### Scenario: UI для текстового типа
- **WHEN** пользователь выбирает характеристику типа `text`
- **THEN** UI ДОЛЖЕН показать `<textarea>`

#### Scenario: UI для boolean типа
- **WHEN** пользователь выбирает характеристику типа `boolean`
- **THEN** UI ДОЛЖЕН показать выпадающий список с вариантами «Да» (1) и «Нет» (0)

#### Scenario: UI для selectable типов
- **WHEN** пользователь выбирает характеристику типа `select` или `radio` с флагом `selectable=1`
- **THEN** UI ДОЛЖЕН показать выпадающий список с доступными значениями из БД

#### Scenario: Переключение виджета при смене характеристики
- **WHEN** пользователь меняет выбранную характеристику в выпадающем списке
- **THEN** JS ДОЛЖЕН скрыть текущий виджет ввода и показать виджет, соответствующий типу новой характеристики

### Requirement: Доступные значения selectable-характеристик
Backend MUST предзагружать и передавать в шаблон доступные значения для selectable-характеристик.

#### Scenario: Предзагрузка значений
- **WHEN** backend action загружает список характеристик
- **THEN** для каждой характеристики с `selectable=1` сервис ДОЛЖЕН загрузить значения из соответствующей таблицы `shop_feature_values_*` и передать в шаблон как JSON-объект

#### Scenario: Безопасность SQL
- **WHEN** загружаются доступные значения характеристики
- **THEN** запрос ДОЛЖЕН использовать параметризованный SQL с `feature_id` как integer и whitelist таблицы значений

### Requirement: Локализация типов характеристик
Все новые пользовательские тексты MUST быть локализованы на `ru_RU` и `en_US`.

#### Scenario: Новые i18n-ключи
- **WHEN** реализуется типо-зависимый UI
- **THEN** MUST быть добавлены ключи: `feature_type_boolean_yes`, `feature_type_boolean_no`, `feature_type_color_hint`, `feature_type_dimension_hint`, `feature_type_range_hint` в обе локали
