## Purpose

Зафиксировать безопасный поток массовых операций над товарами, ограничения операций и обязательные серверные гарантии.

## Requirements

### Requirement: Безопасный поток применения
Массовая операция MUST проходить через выбор товаров, параметры операции, явное модальное подтверждение, серверную повторную валидацию, батчевое применение и запись в журнал.

#### Scenario: Нет подтверждения
- **WHEN** POST-запрос на применение операции не содержит `confirm_apply`
- **THEN** сервис MUST отклонить операцию до записи данных

#### Scenario: Недостаточно прав
- **WHEN** текущий пользователь не является администратором Shop-Script
- **THEN** backend action, JSON controller и сервис операции MUST отклонить запрос

#### Scenario: Слишком много товаров
- **WHEN** количество выбранных товаров превышает `operation_limit`
- **THEN** сервис MUST отклонить операцию и не начинать транзакцию

### Requirement: Транзакционность и логирование
Применение операции MUST выполняться в транзакции, а успешный результат MUST записываться в `shop_masseditor_log`.

#### Scenario: Успешное применение
- **WHEN** операция валидна и товары загружены
- **THEN** сервис MUST выполнить `START TRANSACTION`, применить изменения батчами, записать лог и выполнить `COMMIT`

#### Scenario: Ошибка во время применения
- **WHEN** во время записи возникает исключение
- **THEN** сервис MUST выполнить `ROLLBACK`, пробросить ошибку, а UI MUST скрыть технические детали от пользователя и записать их в `shop/plugins/masseditor.log`

### Requirement: Поддерживаемые операции
Сервис MUST поддерживать только whitelisted операции: `price`, `compare_price`, `visibility`, `availability`, `description`, `tags`, `url`, `stock`, `features`, `categories`.

#### Scenario: Неизвестная операция
- **WHEN** request содержит operation вне whitelist
- **THEN** сервис MUST вернуть локализованную ошибку неизвестной операции

#### Scenario: Цена и compare price
- **WHEN** выполняется операция цены или compare price
- **THEN** сервис MUST поддерживать режимы `set`, `add`, `subtract`, `increase_percent`, `decrease_percent`, округление 0-4 знака и запрет отрицательного результата

#### Scenario: Описание, теги и URL
- **WHEN** выполняются операции контента
- **THEN** сервис MUST валидировать непустые значения, поддерживать режимы описания `replace`/`prepend`/`append`, теги `add`/`remove`/`replace` и URL `regenerate`/`template`

### Requirement: Остатки, характеристики и категории
Операции остатков, базовых характеристик и категорий MUST соблюдать текущие ограничения релиза 1.1.0.

#### Scenario: Остатки без выбранного склада
- **WHEN** операция stock не указывает склад
- **THEN** сервис МОЖЕТ менять общий SKU count только для товаров без складского учета и MUST отклонить товары со складскими остатками

#### Scenario: Базовые характеристики
- **WHEN** операция features устанавливает значение
- **THEN** сервис MUST поддерживать только базовые одиночные типы `varchar`, `text`, `double`, `int`, валидировать числовые значения и не включать SKU-характеристики, множественные значения, цвета, диапазоны и размерности без отдельного change

#### Scenario: Категории
- **WHEN** операция categories добавляет, удаляет или заменяет основную категорию
- **THEN** сервис MUST использовать подтвержденную category ID, поддерживать режимы `add`, `remove`, `replace_main` и не создавать новые категории в рамках массовой операции
