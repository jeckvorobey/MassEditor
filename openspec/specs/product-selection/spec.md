## Purpose

Зафиксировать контракт выборки товаров, фильтров, пагинации, подсказок поиска и пакетной загрузки связанных данных.

## Requirements

### Requirement: Серверная выборка товаров
Список товаров MUST формироваться серверно через `shopMasseditorPluginProductSelectionService` и `shopMasseditorPluginProductModel`, с нормализацией фильтров `query`, `status`, `availability`, `category_id`, `stock_id` и безопасными параметрами SQL.

#### Scenario: Получение страницы товаров
- **WHEN** backend-экран запрашивает список товаров
- **THEN** сервис MUST вернуть `products`, нормализованные `filters` и `pagination` с корректными `page`, `page_size`, `total`, `pages`

#### Scenario: Некорректные фильтры
- **WHEN** входные фильтры содержат неизвестный `status` или `availability`
- **THEN** сервис MUST привести их к `all`

#### Scenario: Некорректный склад
- **WHEN** входные фильтры содержат `stock_id`, которого нет в списке складов Shop-Script
- **THEN** сервис MUST привести `stock_id` к `0` и MUST выбрать товары без ограничения по складу

#### Scenario: Фильтр по складу
- **WHEN** входные фильтры содержат существующий `stock_id`
- **THEN** сервис MUST вернуть только товары, у которых есть SKU со складской записью в `shop_product_stocks` для этого склада
- **AND** SQL MUST использовать плейсхолдеры и не MUST размножать товары из-за нескольких SKU

### Requirement: Расширенный поиск
Поиск MUST поддерживать название, основной SKU, URL, summary, description, теги и категории без прямой склейки пользовательского ввода в SQL.

#### Scenario: Поисковый запрос
- **WHEN** пользователь вводит строку поиска
- **THEN** SQL MUST использовать placeholder-параметры, а связанные таблицы SKU, тегов и категорий MUST подключаться только через сервис выборки

#### Scenario: Поисковые подсказки
- **WHEN** query короче 2 символов
- **THEN** controller подсказок MUST вернуть пустой список без запроса к сервису

#### Scenario: Подсказки с активными фильтрами
- **WHEN** запрашиваются подсказки при активных status, availability, category или stock фильтрах
- **THEN** сервис MUST применить эти фильтры, игнорируя текущий query фильтр страницы, и вернуть уникальные ограниченные значения

### Requirement: Пагинация и лимиты
Пагинация MUST нормализовать номер страницы и размер страницы, чтобы не выходить за доступный диапазон и не делать чрезмерные выборки.

#### Scenario: Page size вне диапазона
- **WHEN** page size не задан или превышает верхнюю границу
- **THEN** сервис MUST использовать безопасное значение: по умолчанию 50, максимум 200

### Requirement: Выбор всех товаров по фильтру
Выбор всех найденных товаров MUST выполняться серверной повторной выборкой ID по текущим фильтрам с учетом лимита операции. На мобильных устройствах (≤1024px) пользователь также ДОЛЖЕН иметь доступ к функции «Выбрать все на странице» через отдельный чекбокс в зоне действий таблицы, аналогичный десктопному чекбоксу в `thead`.

#### Scenario: Filter selection
- **WHEN** пользователь выбирает все товары по текущему фильтру
- **THEN** `getIdsByFilters()` MUST вернуть distinct ID в серверном порядке и ограничить выборку лимитом операции

#### Scenario: Filter selection со складом
- **WHEN** пользователь выбирает все товары по текущему фильтру и активен `stock_id`
- **THEN** `getIdsByFilters()` MUST применить фильтр склада так же, как страница товаров
- **AND** массовая операция MUST получить только ID товаров, подходящих под выбранный склад списка

#### Scenario: Мобильный выбор всех на странице
- **WHEN** пользователь на мобильном устройстве (≤1024px) отмечает чекбокс «Выбрать все на странице»
- **THEN** JS MUST отметить все видимые чекбоксы товаров на текущей странице и сохранить их ID в `localStorage`, аналогично десктопному поведению `data-role="select-all"`

### Requirement: Пакетная загрузка связанных данных
Список товаров MUST загружать категории, теги, SKU и остатки батчами, без N+1 запросов по каждому товару. При выбранном фильтре склада отображаемый остаток MUST относиться только к этому складу; без выбранного склада MUST сохраняться детализация по всем складам.

#### Scenario: Остатки по всем складам без фильтра
- **WHEN** страница содержит товары и фильтр `stock_id = 0`
- **THEN** `attachStockDetails()` MUST одним батчем загрузить остатки по product IDs, сохранить общий `count` товара и показать `∞`, если хотя бы один складской остаток SKU бесконечный; `stock_details` MUST содержать детали всех складов, участвующих в учете товара

#### Scenario: Остаток выбранного склада
- **WHEN** страница содержит товары и фильтр `stock_id > 0`
- **THEN** сервис MUST одним батчем ограничить агрегацию остатков этим складом, установить отображаемый `count` товара по выбранному складу и вернуть не более одной записи в `stock_details` для выбранного склада

#### Scenario: Нулевой или бесконечный остаток выбранного склада
- **WHEN** у товара есть складская запись для выбранного склада с `count = 0` или хотя бы один SKU имеет `count IS NULL`
- **THEN** сервис MUST вернуть соответственно `0` или `null`, а backend UI MUST показать `0` или `∞` без подмены общим остатком товара

#### Scenario: Скрытие детализации выбранного склада
- **WHEN** backend UI отображает строку товара при `filters.stock_id > 0`
- **THEN** desktop- и mobile-разметка MUST показать остаток выбранного склада и MUST НЕ рендерить кнопку `Склады` и popover с перечнем складов

#### Scenario: Кнопка складов без фильтра
- **WHEN** backend UI отображает строку товара при `filters.stock_id = 0` и у товара есть складские детали
- **THEN** desktop- и mobile-разметка MUST сохранить кнопку `Склады` и popover с деталями всех складов
