# Спецификация разработки для Codex: Shop-Script плагин `masseditor`

## 1. Назначение продукта

Разработать коммерческий backend-only плагин для Webasyst Shop-Script: **«Массовый редактор товаров»**.

Плагин должен позволять администратору магазина быстро выполнять массовые изменения товаров без ручного редактирования каждой карточки.

Первый релиз должен быть компактным, проверяемым и безопасным для публикации в Webasyst Store.

---

## 2. Тип разработки

- Тип: **Shop-Script plugin**
- ID плагина: `masseditor`
- Основное приложение: `shop`
- Frontend-витрина: **не используется в MVP**
- Основная зона: backend Shop-Script
- Методология: **TDD — сначала тест/проверка ожидаемого поведения, затем реализация**

---

## 3. MVP-функциональность

### 3.1. Обязательный функционал первой версии

Плагин должен поддерживать:

1. Отдельный backend-экран плагина.
2. Список товаров с фильтрами.
3. Выбор товаров для массового изменения.
4. Массовое изменение:
   - основной цены;
   - зачеркнутой цены / compare price, если поле доступно в текущей версии Shop-Script;
   - видимости товара;
   - доступности товара;
   - остатков, если структура складского учета это позволяет безопасно сделать.
5. Предпросмотр изменений перед применением.
6. Подтверждение операции перед записью в БД.
7. Лог операций:
   - ID администратора;
   - дата/время;
   - тип операции;
   - количество затронутых товаров;
   - краткое описание изменения.
8. Корректная установка и удаление собственных таблиц.

---

## 4. Функционал НЕ входит в первый MVP

Не реализовывать в первой итерации:

- frontend-блоки;
- интеграции с внешними API;
- cron-задачи;
- импорт/экспорт CSV/XLSX;
- сложные пресеты фильтров;
- rollback изменений;
- поддержку маркетплейсов;
- AI-функции;
- массовое изменение характеристик, если нет подтвержденной безопасной схемы работы с API Shop-Script.

Эти функции можно добавить после стабильного MVP.

## 6. Архитектурные принципы

### 6.1. Backend-only

Первый релиз не должен зависеть от тем дизайна и storefront-хуков. Это снижает риск конфликтов с кастомными темами и ускоряет публикацию.

### 6.2. Минимальное вмешательство в БД

Разрешено создать только собственную таблицу логов плагина.

Не изменять стандартные таблицы Shop-Script без необходимости.

### 6.3. Без хака ядра

Запрещено изменять:

- файлы ядра Webasyst;
- файлы Shop-Script вне директории плагина;
- системные классы платформы.

### 6.4. Совместимость

Плагин должен быть написан так, чтобы не зависеть от legacy UI, если это возможно.

Если конкретный backend-хук или способ встраивания не подтвержден документацией — Codex должен остановиться и явно пометить участок как требующий проверки.

## 7. База данных

### 7.1. Таблица логов

Предлагаемая таблица:

```sql
shop_masseditor_log
```

Поля:

```text
id              int unsigned auto_increment primary key
user_id         int unsigned null
action_type     varchar(64) not null
entity_count    int unsigned not null default 0
description     text null
created_at      datetime not null
```

### 7.2. Правила БД

- Использовать SQL placeholders.
- Все собственные таблицы должны иметь префикс `shop_masseditor_`.
- Все таблицы плагина должны удаляться в `uninstall.php`.
- Не хранить лишние данные.
- Не хранить персональные данные без необходимости.

## 8. TDD-подход

Разработка идет маленькими шагами.

Для каждой функции сначала формулируется проверяемое поведение:

```text
Given: есть товары с ценой 100
When: администратор применяет увеличение цены на 10%
Then: новая цена становится 110
And: операция записывается в лог
```

### 8.1. Минимальный набор проверок

1. Плагин устанавливается без ошибок.
2. Таблица логов создается.
3. Backend-экран открывается.
4. Список товаров загружается.
5. Фильтр товаров возвращает ожидаемый набор.
6. Предпросмотр не изменяет данные в БД.
7. Применение операции изменяет только выбранные товары.
8. После операции создается запись в логе.
9. Удаление плагина очищает собственные таблицы.
10. Ошибочный запрос не приводит к частичному изменению данных.

---

## 9. Требования к безопасности

- Проверять права текущего пользователя перед выполнением операций.
- Все POST-запросы должны проверяться на CSRF, если это предусмотрено Webasyst в данном контексте.
- Не доверять данным из запроса.
- Все ID приводить к integer.
- Все числовые значения валидировать.
- Не выводить сырые пользовательские данные без экранирования.
- Не писать файлы вне разрешенных директорий Webasyst.

---

## 10. Требования к UX

Первый экран должен быть простым:

1. Блок фильтров.
2. Таблица товаров.
3. Панель массового действия.
4. Кнопка «Предпросмотр».
5. Кнопка «Применить изменения» после предпросмотра.
6. Блок последних операций.

Обязательно:

- предупреждение перед массовым изменением;
- отображение количества затронутых товаров;
- понятный текст ошибки;
- отсутствие необратимых действий без подтверждения.

## 11. Этапы разработки

### Этап 1. Каркас плагина

- Создать структуру директорий.
- Создать `plugin.php`.
- Создать основной класс `shopMasseditorPlugin`.
- Создать backend action.
- Создать шаблон backend-страницы.
- Подключить JS/CSS.

Критерий готовности: экран плагина открывается в админке.

### Этап 2. Таблица логов

- Создать модель логов.
- Создать `install.php`.
- Создать `uninstall.php`.
- Добавить метод записи лога.

Критерий готовности: установка создает таблицу, удаление ее очищает.

### Этап 3. Загрузка товаров

- Получить список товаров.
- Добавить базовую пагинацию.
- Добавить фильтры.

Критерий готовности: администратор видит список товаров.

### Этап 4. Предпросмотр изменения цены

- Реализовать расчет изменения без записи в БД.
- Показать старое и новое значение.

Критерий готовности: предпросмотр не меняет данные.

### Этап 5. Применение изменения цены

- Записать изменения в БД через безопасный механизм.
- Добавить лог операции.
- Обработать ошибки.

Критерий готовности: меняются только выбранные товары, операция логируется.

### Этап 6. Видимость и доступность

- Добавить массовое изменение видимости.
- Добавить массовое изменение доступности.

Критерий готовности: выбранные товары меняют статус.

### Этап 7. Подготовка к публикации

- Проверить PHP/JS ошибки.
- Проверить uninstall.
- Добавить README.
- Добавить краткую документацию.
- Проверить работу на тестовом магазине.

## 12. Ограничения для Codex

Codex не должен:

- придумывать несуществующие API Webasyst;
- использовать неподтвержденные хуки;
- менять ядро;
- писать монолитный файл вместо структуры плагина;
- реализовывать функции вне текущего этапа;
- создавать frontend-часть без отдельного задания;
- добавлять Composer-зависимости без разрешения;
- делать массовые SQL-запросы без предварительного предпросмотра и валидации.

---

## 13. Definition of Done

Функция считается готовой, если:

- есть проверяемый сценарий поведения;
- код минимален и не дублируется;
- нет PHP syntax errors;
- нет JS console errors в основном сценарии;
- есть обработка ошибок;
- нет прямого вывода неэкранированных данных;
- операция безопасна для большого каталога;
- изменения не выходят за рамки текущего этапа.
