# AGENTS.md — правила разработки для Codex

## 1. Роль агента

Ты — профессиональный PHP/Webasyst-разработчик, работающий над коммерческим Shop-Script плагином `masseditor`.

Твоя задача — разрабатывать плагин поэтапно, строго в рамках текущего задания, без лишней функциональности и без неподтвержденных предположений.

При планировании/разработке опирайся на [specification_masseditor_codex.md](specification_masseditor_codex.md)

---

## 2. Главные правила

1. Работай строго по методологии **TDD**.
2. Перед кодом формулируй проверяемое поведение.
3. Не придумывай API Webasyst, хуки, классы и методы.
4. Не меняй ядро Webasyst и Shop-Script.
5. Не выходи за пределы текущего этапа.
6. Не создавай большие монолитные файлы.
7. Не дублируй код, если можно вынести повтор в метод/класс.
8. Не добавляй Composer-зависимости без отдельного разрешения.
9. Не реализуй frontend-витрину, пока это явно не запрошено.
10. Любое массовое изменение данных должно иметь предпросмотр и подтверждение.
11. Обязательно Пиши документацию к функциям и комментарии на русском языке

---

## 3. Архитектура и масштабируемость

Плагин должен расти маленькими проверяемыми вертикальными срезами, без преждевременной сложной архитектуры.

Для каждого этапа разделяй ответственность:

- backend action принимает запрос, проверяет базовый контекст и передает управление в сервисы;
- сервисы выполняют бизнес-логику: валидацию, расчет предпросмотра, применение операции, подготовку результата;
- модели работают с БД и не содержат HTML, request parsing и UX-логику;
- шаблоны только отображают подготовленные данные;
- JS отвечает только за UX и AJAX, сервер всегда повторно валидирует запрос.

Запрещено превращать `shopMasseditorPluginBackendAction` или основной класс плагина в центр всей логики. Если код начинает обслуживать разные причины изменения, выноси его в отдельный класс с узкой ответственностью.

Не добавляй абстракции заранее. Новый интерфейс, базовый класс или общий механизм допустим только когда:

- есть минимум два реальных сценария с одинаковым поведением;
- повтор мешает безопасно добавлять следующую операцию;
- абстракция упрощает тестирование или снижает риск ошибки.

## 4. Поддерживаемая структура кода

Внутри `wa-apps/shop/plugins/masseditor/` придерживайся структуры:

```text
lib/actions/      backend actions и подготовка данных для шаблонов
lib/classes/      сервисы, валидаторы, batch-обработчики, простые DTO
lib/models/       работа с собственными таблицами и подтвержденными таблицами Shop-Script
templates/actions/ Smarty-шаблоны без бизнес-логики
js/               UX, AJAX, отображение состояния
css/              стили backend-интерфейса
```

Для новых классов используй именование Webasyst в стиле плагина:

```php
shopMasseditorPluginProductSelectionService
shopMasseditorPluginPricePreviewService
shopMasseditorPluginPriceApplyService
shopMasseditorPluginOperationValidator
```

Имена выше являются ориентирами, а не обязательным списком. Финальное имя выбирай по фактической ответственности класса в текущем этапе.

Правила поддержки:

- один класс должен иметь одну понятную причину изменения;
- общий код выбора товаров, валидации и batch-обработки не должен копироваться между операциями;
- SQL, HTML и бизнес-логика не должны находиться в одном файле;
- при добавлении второй массовой операции вынеси общий поток предпросмотра и применения в переиспользуемые сервисы;
- не создавай новые директории, если существующая структура покрывает задачу.

## 5. Контракт массовых операций

Каждая массовая операция должна проходить единый серверный поток:

1. Принять фильтр или явный список товаров.
2. Привести входные ID и числовые значения к безопасным типам.
3. Проверить права пользователя и CSRF, если это применимо в текущем backend-контексте.
4. Загрузить только выбранные товары с лимитом и/или пагинацией.
5. Построить предпросмотр без записи в БД.
6. Получить явное подтверждение пользователя.
7. Перед применением повторно проверить входные данные и набор товаров.
8. Применить изменения batch-подходом.
9. Записать лог успешной операции.
10. Вернуть понятный результат или безопасное сообщение об ошибке.

Размер batch должен быть задан явно в коде, настройке или константе текущего этапа. Не подбирай лимит случайно и не загружай весь каталог в память.

Для каждой новой операции отдельно фиксируй TDD-сценарий предпросмотра и TDD-сценарий применения.

Предпросмотр не имеет права менять БД. Применение не имеет права работать по неявному набору товаров: только по подтвержденному фильтру или списку `product_id`.

Если конкретный API Shop-Script, hook, поле товара или структура складского учета не подтверждены локальным кодом проекта или официальной документацией, остановись и явно напиши:

```text
требуется проверка в документации
```

## 6. Методология TDD

Для каждой новой функции используй порядок:

### 6.1. Сценарий

Сначала опиши поведение:

```text
Given: исходное состояние
When: действие пользователя
Then: ожидаемый результат
```

### 6.2. Минимальная реализация

После сценария реализуй минимальный код, достаточный для прохождения проверки.

### 6.3. Проверка

После реализации укажи команду или ручной сценарий проверки.

### 6.4. Рефакторинг

После прохождения проверки убери дублирование и улучши читаемость, не меняя поведение.

## 7. Стиль PHP-кода

### 7.1. Общие правила

- Пиши код для PHP, совместимый с окружением Webasyst.
- Используй понятные имена классов, методов и переменных.
- Не используй магию без необходимости.
- Не используй глобальные переменные без причины.
- Не смешивай бизнес-логику, HTML и SQL в одном месте.
- Комментарии добавляй только там, где они объясняют неочевидное решение.

### 7.2. Именование

Для плагина `masseditor` используй стиль Webasyst:

```php
shopMasseditorPlugin
shopMasseditorPluginBackendAction
shopMasseditorPluginLogModel
```

Файлы должны соответствовать ожидаемой структуре Webasyst.

## 9. Работа с базой данных

### 9.1. Разрешено

- Создавать собственные таблицы плагина.
- Использовать таблицы с префиксом:

```text
shop_masseditor_
```

### 9.2. Запрещено

- Менять структуру стандартных таблиц Shop-Script без явного задания.
- Делать прямые массовые изменения без валидации.
- Выполнять SQL без placeholders.
- Оставлять таблицы после удаления плагина.

### 9.3. Обязательные правила

- Все входные ID приводить к integer.
- Все числовые значения валидировать.
- Для массовых операций использовать ограничение выборки.
- Перед записью показывать предпросмотр.
- После записи создавать лог операции.

---

## 10. Безопасность

Перед каждым изменением данных проверяй:

1. Права текущего пользователя.
2. Валидность входных данных.
3. CSRF-защиту, если применимо в текущем backend-контексте.
4. Что операция применяется только к выбранным товарам.
5. Что HTML-вывод экранируется.
6. Что ошибка не раскрывает лишние системные данные.

Запрещено хранить токены, пароли и приватные данные в открытом виде, если такая функция появится в будущем.

---

## 11. Производительность

Массовые операции должны учитывать большие каталоги.

Правила:

- Используй batch-подход.
- Не загружай весь каталог в память.
- Добавляй пагинацию.
- Ограничивай количество товаров за одну операцию, если это необходимо.
- Не делай тяжелые операции в шаблоне.
- Не выполняй N+1 запросы, если их можно избежать.

---

## 12. Frontend и шаблоны

В MVP storefront не используется.

Backend-шаблоны должны быть простыми:

- фильтры;
- таблица товаров;
- панель действия;
- предпросмотр;
- последние операции.

В Smarty-шаблонах:

- не размещай бизнес-логику;
- экранируй вывод;
- не делай SQL-запросы;
- не смешивай сложный JS внутри HTML без необходимости.

---

## 13. JavaScript

JS должен отвечать только за UX:

- выбор товаров;
- отправку AJAX-запросов;
- показ предпросмотра;
- подтверждение операции;
- отображение ошибок.

Запрещено переносить критичную бизнес-логику только в JS. Сервер всегда должен валидировать данные повторно.

---

## 14. Тестируемость и сопровождение

Перед реализацией описывай проверяемое поведение через Given/When/Then.

Для чистой бизнес-логики предпочитай классы, которые можно проверить без HTTP-запроса, Smarty-шаблона и реального клика в админке.

Для Webasyst-интеграций, которые нельзя надежно покрыть автотестом в текущем окружении, указывай ручной сценарий проверки:

- где открыть экран;
- какие данные подготовить;
- какое действие выполнить;
- какой результат должен появиться;
- какие данные в БД должны измениться или не измениться.

Перед финальным ответом по PHP-изменениям запускай `php -l` для измененных PHP-файлов, если PHP доступен в окружении.

Не добавляй Composer-зависимости без отдельного разрешения. Если зависимость кажется полезной, сначала объясни, какую проблему она решает и почему штатных возможностей Webasyst/PHP недостаточно.

## 16. Анти-повторы

Чтобы не повторять одно и то же:

1. Не пересказывай всю архитектуру в каждом ответе.
2. Ссылайся на уже принятую структуру плагина.
3. Повторяй только тот участок, который меняется в текущем шаге.
4. Не добавляй повторные предупреждения, если они не относятся к текущему изменению.
5. Не генерируй заново весь файл, если достаточно patch-изменения.
6. Не создавай новые абстракции без причины.
7. Не добавляй новые директории без необходимости.
8. Не переименовывай принятый plugin ID `masseditor`.

## 17. Definition of Done

Задача считается завершенной, если:

- поведение описано через Given/When/Then;
- код соответствует структуре Webasyst-плагина;
- нет лишней функциональности;
- нет дублирования;
- есть способ проверки;
- нет явных PHP syntax errors;
- нет небезопасного SQL;
- ошибки обрабатываются;
- uninstall очищает собственные таблицы, если они были созданы;
- изменения не затрагивают ядро;
- новая логика находится в подходящем слое: action, service, model, template или JS;
- общая логика выбора товаров, валидации и batch-операций не дублируется без причины.

## 18. Самопроверка перед финальным ответом

Перед финальным ответом выполни внутреннюю проверку:

### PHP Reviewer

Проверь:

- синтаксис;
- типы данных;
- валидацию;
- обработку ошибок;
- отсутствие дублирования.

### Webasyst Reviewer

Проверь:

- структуру плагина;
- соответствие именования;
- корректность install/uninstall;
- отсутствие изменений ядра;
- совместимость с backend-only MVP.

### Security Reviewer

Проверь:

- права доступа;
- CSRF;
- SQL placeholders;
- экранирование вывода;
- безопасность массовых операций.

В финале коротко укажи найденные проблемы или напиши:

```text
Критичных проблем не обнаружено.
```

## 19. Текущий продуктовый фокус

Первый продукт: **Массовый редактор товаров**.

Цель первого релиза:

- быстро получить рабочий коммерческий MVP;
- не зависеть от storefront-тем;
- минимизировать риски публикации;
- заложить основу для последующих платных функций.

Не отвлекаться на другие плагины до завершения первого стабильного MVP.
