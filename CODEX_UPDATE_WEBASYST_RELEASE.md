# Задача для Codex CLI: обновить plugin/skill `webasyst-release`

Работай как senior release engineer для Webasyst и Shop-Script. Обнови существующий Codex plugin/skill `webasyst-release`, чтобы он воспроизводимо готовил полную релизную документацию и архив Webasyst-плагина в едином формате.

## Цель

После изменения `webasyst-release` должен для любой новой версии Webasyst-плагина:

1. определить фактические возможности продукта и изменения относительно предыдущей опубликованной версии;
2. обновить версию в конфигурации продукта;
3. сформировать описание Webasyst Store со всеми актуальными возможностями;
4. сформировать release note только с изменениями новой версии;
5. обновить полный changelog без потери предыдущих версий;
6. выполнить это для всех языков, поддерживаемых конкретным плагином;
7. проверить локализацию и пересобрать `.mo`, если они входят в продукт;
8. проверить необходимость meta-update и его идемпотентность;
9. запустить проверки и тесты проекта;
10. создать или атомарно пересобрать архив нужной версии;
11. проверить структуру и содержимое архива;
12. сформировать итоговый отчёт и чеклист ручной публикации;
13. не публиковать продукт и не создавать скриншоты автоматически.

## Обязательный порядок работы

### 1. Диагностика

Сначала прочитай:

- корневой и вложенные `AGENTS.md`;
- правила проекта и SSOT-документы;
- текущую реализацию `webasyst-release`;
- его команды, templates, scripts, tests и README;
- структуру целевого Webasyst-продукта;
- `lib/config/plugin.php`;
- каталоги `locale/`, `lib/updates/`, `tests/`, `dist/` и текущие release-документы;
- Git history и теги, чтобы определить предыдущую опубликованную версию.

Не начинай реализацию, пока не опишешь текущий workflow и найденные проблемы.

### 2. Официальные источники

Проверь актуальные официальные страницы Webasyst и зафиксируй дату проверки:

- https://developers.webasyst.com/docs/store/webasyst-store-requirements/
- https://developers.webasyst.com/docs/store/product-update/
- https://developers.webasyst.com/docs/store/check-list/
- https://developers.webasyst.com/docs/features/console-tools/
- https://developers.webasyst.com/docs/cookbook/meta-updates/
- https://developers.webasyst.com/docs/cookbook/plugins/

Не копируй документацию целиком. В README или reference-файле skill сохрани только применяемые правила и ссылки на первичные источники.

### 3. Аудит текущей документации

На примере Mass Editor проверь как минимум:

- `store-description-1_0_0.md`;
- `store-description-1_1_0.md`;
- `store-description-1_2_0.md`;
- `release-1.1.0.md`;
- `release-1.1.1.md`;
- `release-1.2.0.md`;
- `CHANGELOG.md`;
- `CHANGELOG.en.md`;
- `store-publication.md`.

Исправь системные причины следующих проблем, а не только эти файлы:

- описание Store представляет функции плоским списком и не повторяет структуру интерфейса;
- release note новой версии может повторять изменения предыдущей версии;
- patch-версия `1.1.1` существует отдельным документом, но отсутствует в changelog;
- разные документы вручную дублируют один набор фактов и могут расходиться;
- в публичные тексты попадают внутренние детали вроде Docker-порта и рефакторинга без пользовательского эффекта;
- список операций в разделе «Как это работает» устаревает после добавления новых функций;
- языки могут быть жёстко заданы вместо определения из продукта;
- архив проверяется как копия всего source tree, хотя официальный упаковщик может исключать dev-файлы.

### 4. Единая модель данных

Интегрируй решение в существующую архитектуру plugin/skill. Не создавай параллельный второй workflow.

Если у `webasyst-release` нет единого источника данных, добавь минимальную структурированную модель без тяжёлых зависимостей:

- конфигурация продукта: ID, slug, локали, пути, названия, backend-only, текущие ограничения;
- каталог актуальных возможностей с группами интерфейса и `since_version`;
- отдельный манифест каждой версии с delta-изменениями и доказательствами;
- JSON Schema или эквивалентная валидация.

Предпочитай JSON и стандартную библиотеку языка, если YAML потребует новую runtime-зависимость. Если в skill уже есть устойчивый формат, расширь его вместо миграции ради миграции.

Модель должна быть идемпотентной: повторный запуск для той же версии обновляет существующие артефакты без дубликатов.

### 5. Доказательность функций

Для каждого пункта, попадающего в описание Store, создай проверяемую связь минимум с одним источником:

- реализация в коде;
- существующий тест;
- новый тест;
- подтверждённая ручная проверка;
- явно отмеченное ограничение.

Не считай скриншот достаточным доказательством поведения. Например, пункт «Откат» в интерфейсе не означает, что можно обещать полный откат любой операции.

Перед добавлением «отмены массовой операции» проверь:

- какие операции поддерживают откат;
- восстанавливаются ли все затронутые поля;
- есть ли ограничения по времени и журналу;
- что происходит после последующих изменений товара;
- есть ли серверная проверка прав и целостности;
- существуют ли тесты.

Если это не подтверждено, не добавляй обещание в Store, а внеси пункт в список неподтверждённых возможностей.

### 6. Стандарт описания Store

Описание текущей версии должно быть кумулятивным: содержать все реально доступные возможности продукта.

Порядок разделов для Mass Editor должен повторять интерфейс:

1. Что нового в текущей версии.
2. Цены и остатки:
   - изменение цены;
   - изменение цены сравнения;
   - остатки.
3. Контент:
   - видимость;
   - доступность;
   - описание;
   - теги;
   - категории.
4. Медиа:
   - видео.
5. URL и страницы:
   - URL товаров.
6. Параметры и характеристики:
   - редактирование характеристик.
7. Выбор товаров и фильтры.
8. Контроль массовых операций:
   - подтверждение;
   - лимиты;
   - серверная валидация;
   - пакетная обработка;
   - журнал;
   - откат, только если полностью подтверждён.
9. Как это работает.
10. Ограничения.
11. Backend-only формулировка.

Каждая операция описывается коротко: название, пользовательский результат, основные режимы, критичное ограничение.

Раздел «Как это работает» не должен содержать вручную поддерживаемый перечень операций. Используй универсальную формулировку «выберите нужную операцию».

Для Store HTML используй только базовые теги, уже принятые в проекте: `p`, `strong`, `ul`, `ol`, `li`, `code`, `blockquote`. Не добавляй CSS или JavaScript.

### 7. Release note

Release note содержит только изменения конкретной версии:

- Кратко;
- Добавлено;
- Изменено;
- Исправлено;
- Безопасность;
- Ограничения версии;
- Обновление установленного плагина;
- Проверка релиза.

Пустые разделы не выводятся.

Публичная заметка для Store не должна включать:

- Docker-порты;
- имена внутренних методов;
- рефакторинг без пользовательского эффекта;
- тестовые и локальные изменения;
- изменения, уже выпущенные в предыдущей версии.

Технические детали можно сохранить в engineering report.

### 8. Changelog

Реализуй следующие инварианты:

- новые версии сверху;
- формат версии `MAJOR.MINOR.PATCH`;
- каждая опубликованная версия присутствует ровно один раз;
- нет пропусков между обнаруженными опубликованными release notes/tags;
- дата и набор смысловых изменений совпадают во всех локалях;
- изменение принадлежит только той версии, где оно впервые опубликовано;
- фиксированный порядок категорий: Added, Changed, Fixed, Security, Deprecated, Removed;
- пустые категории отсутствуют;
- повторный запуск не создаёт дубль версии;
- HTML-блок для кабинета Store, если он нужен, генерируется из тех же данных.

Добавь автоматическую проверку, которая обнаружит ситуацию, аналогичную отсутствию `1.1.1` в текущем changelog.

### 9. Языки и gettext

Определи поддерживаемые локали по фактической структуре и конфигурации продукта. Не ограничивайся `ru_RU` и `en_US`.

Для каждой локали:

- сформируй Store description;
- сформируй release note;
- обнови changelog;
- проверь одинаковые версии, даты, структуру и количество смысловых пунктов;
- проверь отсутствие незаполненных placeholders и непереведённых обязательных секций.

Для UI gettext:

1. используй штатную команду `php wa.php locale <slug> --debug`, если Webasyst CLI доступен;
2. проверь новые и неиспользуемые ключи вручную/скриптом;
3. не удаляй ключи, которые формируются динамически, без доказательства;
4. пересобери `.mo` из `.po`, если `.mo` входит в публикационный архив;
5. заблокируй релиз при ошибке компиляции или отсутствующем переводе пользовательской строки.

### 10. Версия и meta-updates

Проверяй версию по официальному принципу Webasyst:

- PATCH — исправления и небольшие улучшения;
- MINOR — новая функциональность или крупное улучшение;
- MAJOR — значительная переработка продукта.

Версия в `lib/config/plugin.php`, release manifest, документах, changelog и архиве должна совпадать.

Проверяй необходимость meta-update, если:

- добавлены или изменены таблицы/поля;
- изменён формат хранимых данных;
- из новой версии удалены runtime-файлы предыдущей версии.

Meta-update должен:

- находиться в `lib/updates/`;
- иметь корректное timestamp-имя;
- быть безопасным при повторном выполнении;
- не прерывать цепочку обновлений из-за ожидаемой повторной операции;
- иметь тест обновления с предыдущей опубликованной версии.

Не создавай собственную параллельную таблицу миграций, если проект использует штатные meta-updates Webasyst.

### 11. Проверки релиза

Сохрани существующие project-specific тесты и добавь orchestration поверх них.

Минимально выполни:

- PHP lint для runtime PHP-файлов;
- JS syntax/lint по конфигурации проекта;
- unit/integration tests;
- проверку локализации;
- проверку version consistency;
- проверку changelog completeness;
- проверку parity между локалями;
- проверку meta-update;
- тест обновления с предыдущей опубликованной версии, если среда позволяет;
- проверку отсутствия секретов и запрещённых файлов в distribution set.

Не скрывай провалы тестов. Статус `READY` допустим только при успешных обязательных проверках и явном списке оставшихся ручных действий.

### 12. Архив

Основной способ упаковки:

```bash
php wa.php compress <app_id>/plugins/<plugin_id>
```

Для Mass Editor slug:

```bash
php wa.php compress shop/plugins/masseditor
```

Не используй `-skip test` в обычном релизном режиме.

После выполнения:

1. найди созданный архив по выводу команды или фактическому расположению, не угадывай путь;
2. атомарно помести итоговый артефакт в `dist/<plugin_id>-<version>.tar.gz`, если это соответствует текущему проектному соглашению;
3. выполни `gzip -t`;
4. выполни `tar -tzf`;
5. проверь, что в корне ровно один каталог `<plugin_id>/`;
6. проверь наличие обязательных runtime-файлов;
7. проверь отсутствие `.git`, `.env`, ключей, дампов, логов, временных файлов и лишних dev-артефактов;
8. распакуй архив во временный каталог и сравни его с ожидаемым distribution manifest, а не обязательно со всем source tree;
9. вычисли SHA-256 и сохрани рядом;
10. удали временные каталоги.

Поведение при существующем архиве:

- если версия ещё не опубликована, разрешено атомарно пересобрать архив после успешных проверок;
- если версия отмечена как опубликованная, запрети перезапись без явного `--force`/подтверждения;
- никогда не оставляй частично созданный архив под финальным именем.

Если Webasyst CLI недоступен, не молча переключайся на самодельный `tar`. Заверши с понятной ошибкой или используй явно разрешённый fallback, который повторяет официальные требования к структуре архива.

### 13. Скриншоты

Скриншоты готовит пользователь.

`webasyst-release` должен только:

- сгенерировать список необходимых экранов на основании новых и основных функций;
- предложить короткие подписи на всех языках;
- при наличии файлов проверить имена, формат, отсутствие экрана логина и максимальную исходную ширину согласно официальным требованиям;
- не генерировать и не подменять реальные backend-скриншоты.

### 14. UX команды/skill

Сохрани совместимость с текущим способом вызова `webasyst-release`. Если нужно расширение интерфейса, добавь понятные параметры, например:

- целевая версия;
- дата релиза;
- статус draft/published;
- путь к продукту;
- режим dry-run;
- разрешение пересборки draft-архива;
- язык/все языки.

Не заставляй пользователя вручную повторять данные, которые можно надёжно определить из репозитория.

В режиме dry-run покажи:

- определённую предыдущую версию;
- предлагаемый version bump;
- обнаруженные изменения;
- документы, которые будут изменены;
- проверки, которые будут выполнены;
- путь будущего архива;
- блокирующие проблемы.

### 15. Тесты самого `webasyst-release`

Добавь fixture-репозитории или минимальные тестовые данные и покрой:

1. MINOR-релиз с новой функцией.
2. PATCH-релиз без новой функции.
3. Пропущенную patch-версию в changelog.
4. Повторный запуск той же версии без дублей.
5. Несовпадение версии между `plugin.php`, документацией и архивом.
6. Дополнительную локаль кроме RU/EN.
7. Неполный перевод.
8. Неидемпотентный или отсутствующий meta-update при изменении БД.
9. Архив с двумя корневыми каталогами.
10. Архив с `.env` или секретом.
11. Попытку перезаписать опубликованный архив.
12. Пользовательскую функцию без доказательства в коде/тесте.
13. Публичный release note с внутренним Docker/refactor-пунктом.
14. Генерацию описания в фиксированном порядке UI-рубрик.

### 16. Документация plugin/skill

Обнови README и встроенные инструкции:

- назначение;
- входные данные;
- последовательность действий;
- создаваемые/изменяемые файлы;
- правила SSOT;
- локализация;
- упаковка;
- dry-run;
- ошибки и блокирующие состояния;
- ручные шаги публикации;
- ссылки на официальную документацию Webasyst;
- пример для Mass Editor.

Добавь отдельный шаблон/standard-файл, эквивалентный `WEBASYST_RELEASE_DOCUMENTATION_STANDARD.md`, либо интегрируй эти правила в существующие references без дублирования.

## Критерии готовности

Работа считается завершённой только если:

- текущий plugin/skill действительно изменён, а не создан параллельный черновик;
- описание Store генерируется в фиксированном порядке рубрик интерфейса;
- основные функции, фильтры, выбор, журнал и подтверждённый откат разделены логически;
- release note содержит только delta конкретной версии;
- changelog содержит все опубликованные версии, включая patch-релизы, и новые версии расположены сверху;
- все локали имеют смысловой parity;
- version consistency проверяется автоматически;
- meta-update проверяется по официальным правилам;
- архив создаётся штатным Webasyst CLI и проходит структурную проверку;
- существующий опубликованный архив защищён от случайной перезаписи;
- тесты `webasyst-release` проходят;
- существующие workflow не сломаны;
- сформирован понятный migration note для уже существующих release-документов;
- итоговый diff просмотрен;
- секреты не добавлены;
- публикация, push и release не выполнены без отдельного разрешения.

## Финальный отчёт

В конце выведи:

1. что изменено в архитектуре `webasyst-release`;
2. какие файлы изменены;
3. какие системные проблемы устранены;
4. какие тесты выполнены и их результат;
5. пример dry-run для Mass Editor;
6. пример списка генерируемых документов;
7. путь и SHA-256 тестового архива, если архив создавался;
8. оставшиеся ручные действия пользователя, включая скриншоты;
9. риски и способ отката изменений plugin/skill.
