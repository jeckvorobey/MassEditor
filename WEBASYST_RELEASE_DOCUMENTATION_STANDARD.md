# Стандарт релизной документации Webasyst-плагинов

## 1. Назначение

Этот стандарт задаёт единый формат для:

- описания карточки Webasyst Store;
- заметки о новой версии;
- полного changelog;
- публикационного чеклиста;
- локализованных вариантов документов;
- архива релиза.

Описание продукта и история версий — разные документы:

- **Описание Store** показывает все актуальные возможности текущей версии.
- **Release note** показывает только изменения конкретной версии.
- **Changelog** хранит историю всех опубликованных версий без пропусков и дублирования изменений между версиями.

## 2. Единый источник данных

Рекомендуется хранить два типа структурированных данных:

1. **Каталог возможностей продукта** — текущее состояние продукта:
   - группы интерфейса;
   - операции;
   - фильтры;
   - управление выбором;
   - журнал и откат;
   - настройки;
   - ограничения;
   - поддерживаемые локали;
   - версия, в которой возможность появилась.

2. **Манифест конкретного релиза** — только изменения версии:
   - версия и дата;
   - тип изменения версии;
   - Added / Changed / Fixed / Security / Removed;
   - изменения данных и meta-update;
   - изменения локализации;
   - тесты;
   - ручные проверки;
   - статус публикации.

Документы Markdown/HTML и changelog должны генерироваться из этих данных, а не редактироваться независимо друг от друга.

## 3. Формат описания Webasyst Store

Порядок разделов фиксированный.

### 3.1. Заголовок

Короткое название продукта без перечисления возможностей.

Пример:

```text
Mass Editor для Shop-Script
```

### 3.2. Краткое описание

Одно или два предложения: назначение, целевой пользователь, область работы продукта.

Не перечислять весь функционал через запятую.

### 3.3. Вводный блок

Кратко объяснить пользовательский сценарий:

1. найти товары;
2. выбрать товары;
3. выбрать операцию;
4. проверить параметры;
5. подтвердить изменение;
6. проверить запись в журнале.

### 3.4. Что нового в текущей версии

Короткий блок только с пользовательскими изменениями последней версии. Не повторять полный changelog и не включать внутренний рефакторинг, Docker-порты или детали реализации, не влияющие на пользователя.

### 3.5. Основные возможности

Для «Массового редактора» порядок должен повторять рубрики интерфейса.

#### Цены и остатки

- **Изменение цены** — короткое описание поддерживаемых режимов.
- **Изменение цены сравнения** — короткое описание режимов.
- **Остатки** — установка, увеличение, уменьшение, бесконечный остаток и подтверждённые варианты складского учёта.

#### Контент

- **Видимость** — опубликован, скрыт, неопубликован.
- **Доступность** — доступность артикулов.
- **Описание** — замена, добавление в начало или конец.
- **Теги** — добавление, удаление, замена.
- **Категории** — подтверждённые режимы управления категориями.

#### Медиа

- **Видео** — подтверждённые способы установки или очистки видео.

#### URL и страницы

- **URL товаров** — генерация и доступные переменные шаблона.

#### Параметры и характеристики

- **Редактирование характеристик** — поддерживаемые типы, одиночные и множественные значения, а также точные ограничения.

Каждый пункт должен содержать:

- название функции;
- одно короткое описание результата для пользователя;
- основные режимы;
- важное ограничение, только если без него описание вводит в заблуждение.

### 3.6. Выбор товаров и фильтры

Отдельный раздел после основных операций:

- поиск по названию и артикулу, если подтверждён;
- статус;
- категория;
- доступность;
- склад;
- тип складского учёта, если подтверждён;
- выбор текущей страницы;
- выбор всех товаров по фильтру;
- сохранение выбора между страницами, если подтверждено.

### 3.7. Контроль массовых операций

Отдельный раздел:

- обязательное подтверждение;
- лимит количества товаров;
- серверная повторная проверка выборки;
- пакетная обработка;
- журнал операций;
- отмена или откат операции — **только после подтверждения фактического поведения по коду, тестам и интерфейсу**;
- права доступа.

Для отката обязательно описать границы: какие операции откатываются, полностью или частично, какой срок хранения данных и что происходит при конфликте с последующими изменениями.

### 3.8. Как это работает

Не более пяти шагов. В списке операций использовать формулировку «выберите нужную операцию», а не перечислять устаревший набор функций вручную.

### 3.9. Ограничения

Только актуальные ограничения текущей версии. Не перечислять гипотетические будущие функции.

### 3.10. Backend-only

Фиксированная формулировка:

```text
Плагин работает только в backend Shop-Script и не влияет на витрину магазина, корзину и оформление заказов.
```

Расширять её только фактами, подтверждёнными кодом.

## 5. Формат release note

Release note описывает только конкретную версию.

```markdown
# {{product_name}} {{version}}

## Кратко

{{one_paragraph_user_value}}

## Добавлено

- {{added_item}}

## Изменено

- {{changed_item}}

## Исправлено

- {{fixed_item}}

## Безопасность

- {{security_item}}

## Ограничения версии

- {{limitation_relevant_to_this_release}}

## Обновление установленного плагина

- {{meta_update_or_no_data_migration}}

## Проверка релиза

- {{automated_test}}
- {{manual_test}}
```

Пустые разделы не выводятся. В публичную заметку Store не включаются внутренние изменения без пользовательского эффекта. Технические изменения остаются в полном release note или engineering report.

## 6. Формат changelog

Правила проекта:

- новые версии располагаются сверху;
- версия имеет формат `MAJOR.MINOR.PATCH`;
- каждая опубликованная версия присутствует ровно один раз;
- дата одинакова во всех языках;
- изменения не переносятся в следующую версию повторно;
- порядок категорий фиксированный: Added, Changed, Fixed, Security, Deprecated, Removed;
- пустые категории не выводятся;
- смысловые пункты совпадают во всех локалях;
- внутренние изменения отделяются от пользовательских.

Пример:

```markdown
## 1.2.0 — 2026-07-14

### Added

- Добавлено массовое редактирование ...

### Changed

- Улучшено ...

### Fixed

- Исправлено ...
```

Если Webasyst Store требует HTML-блок, он генерируется из тех же данных и хранится рядом, а не редактируется отдельно.

## 7. Локализация

Для каждого языка, реально поддерживаемого плагином:

- создаётся описание Store;
- создаётся release note;
- обновляется changelog;
- сохраняется одинаковая версия, дата, структура и набор фактов;
- проверяются `.po`;
- при наличии `.mo` они пересобираются из соответствующих `.po`;
- отсутствие перевода блокирует готовность релиза.

Список языков определяется по конфигурации и каталогам локализации продукта, а не жёстко задаётся как RU/EN.

## 8. Архив релиза

Обязательные правила:

- формат `tar.gz`;
- в корне архива ровно один каталог с ID продукта;
- версия в `lib/config/plugin.php` совпадает с версией документации и именем артефакта;
- архив создаётся штатной командой Webasyst `php wa.php compress <slug>`, если она доступна в проекте;
- архив не содержит `.git`, секретов, временных файлов, логов, дампов и лишних dev-файлов;
- архив проходит `gzip -t` и `tar -tzf`;
- архив распаковывается во временный каталог и проверяется по ожидаемому distribution manifest;
- вычисляется SHA-256;
- опубликованный архив не перезаписывается без явного разрешения.

Нельзя требовать нулевой diff с исходным каталогом плагина, если официальный упаковщик намеренно исключает dev-файлы. Сравнение выполняется с ожидаемым составом дистрибутива.

## 9. Проверки перед публикацией

Минимальный pipeline:

1. Проверить рабочее дерево и определить предыдущую опубликованную версию.
2. Проверить трёхкомпонентную версию в `plugin.php`.
3. Проверить соответствие типа версии фактическим изменениям.
4. Проверить каталог возможностей по коду, интерфейсу и тестам.
5. Проверить, нужны ли meta-updates для БД, формата данных или удаления старых файлов.
6. Проверить идемпотентность meta-updates.
7. Обновить `.po`, переводы и `.mo`.
8. Запустить PHP/JS lint и тесты проекта.
9. Проверить обновление с предыдущей версии на тестовом стенде.
10. Обновить описание, release notes, changelog и публикационный чеклист на всех языках.
11. Создать архив штатной командой Webasyst.
12. Проверить структуру и содержимое архива.
13. Подготовить список реальных скриншотов; сами скриншоты добавляет пользователь.
14. Сформировать итоговый release report.

## 10. Запрещённые действия автоматизации

- Не придумывать возможности по названию файла или элементу интерфейса.
- Не публиковать продукт в Store автоматически.
- Не создавать искусственные скриншоты интерфейса.
- Не менять цену продукта без явного задания.
- Не удалять старые release notes и changelog.
- Не пропускать patch-версии.
- Не перезаписывать опубликованный архив.
- Не объявлять релиз готовым при незаполненной локализации или проваленных тестах.
